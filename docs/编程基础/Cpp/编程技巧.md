# 编程技巧

## 变量

### auto

#### 初始化

使用 `auto` 声明变量，当变量未初始化时就会报错，这样可以避免未初始化的情况

```cpp
// auto 初始化
auto i = int(10);
```

对于具有复杂构建方式的变量，可以使用 lambda 表达式调用来封装初始化过程

```cpp
// lambda 初始化
auto j = [&]() {
    if(i > 0) {
        return 1;
    } else {
        return 2;
    }
}();
```

现在建议使用 `{}` 初始化

```cpp
auto i = int{10};
auto j = int{};
```



#### 相关变量

对于具有固定关系的变量，可以使用 lambda 表达式来构建新的变量。例如

```cpp
int count = 1;
auto count2 = [&]() { return 2 * count; };
```

这样在修改 `count` 时，对应的 `count2` 也会修改，避免出现忘记修改的问题。



### 数组

#### 动态数组

当需要创建临时可变数组时，最好使用 `vector` 而不是 `new` 一个数组，防止内存泄漏问题。如果需要使用数组，可以获得指针

```cpp
auto v = std::vector<int>(10, 0);
v.data();	// 获得数组指针
```



#### 数组大小

由于在数组索引时，当数组指针宽度与索引类型宽度不同时，需要编译器手动扩展，造成性能浪费。例如

```cpp
int func(int *a, int b) {
	return a[b];
}
```

其中 `a` 占 8 字节，`b` 占 4 字节，因此需要先将 `b` 扩充为 8 字节，然后相加。改为使用

```cpp
int func(int *a, std::size_t b) {
	return a[b];
}
```

由于 `std::size_t` 会根据操作系统位数改变宽度，因此不需要额外扩充，更高效。并且它可以容纳超过 `INT_MAX` 的值，推荐作为数组大小和索引。



## 函数

### 返回指针

如果需要返回指针，最好使用 `unique_ptr` 包装

```cpp
std::unique_ptr<int> createInt() {
    return std::make_unique<int>(42);
}
```



### 多参数

对于具有多个参数的函数，可以考虑使用结构体

```cpp
float lerp(float from, float to, float frac) {
	return frac * (to - from) + from;
}

// 使用结构体替代函数 C++ 20
struct lerp {
    float frac = 0.5;
    float to = 1;
    float from = 0;

    // 还可以指定多个返回值
    struct result {
        float value;
        int tag;
    };

    result operator()() { return {frac * (to - from) + from, 1}; }
};

// 避免调整参数顺序导致出错
auto [x, y] = lerp{.frac = 0.4, .to = 1, .from = 0}();
```



### 局部函数

如果需要在源文件中定义临时使用的函数，应当使用 `static` 或匿名命名空间

```cpp
static void test() {

}

namespace {
	void foo() {
	
	}
}
```

由 `static` 修饰的函数或包含在 `namespace` 中的函数只在文件内部可见，因此避免了多个源文件中定义同名函数的冲突。



### 参数类型优化

考虑参数何时需要 `const &` 何时不需要

- 基础类型按值传递（引用、指针反而更大）
- 原始指针按值传递
- 数据容器按常引用传递
- 如果容器不大可以按值传递（例如 `tuple<int, int>`）
- 智能指针且需要生命周期控制，则按值传递
- 智能指针但不需要生命周期控制，则 `.get()` 传递



### 递归函数

图搜索中的深度优先搜索可以使用非递归方式完成。首先需要用栈记录搜索状态，每次使用 `visit` 搜索时，记录搜索路径，然后原路返回

```cpp
std::vector<std::tuple<float, float>> Graph_Sparse::dfs(std::size_t s)
{
    std::vector<std::tuple<float, float>> intervals;
    for (std::size_t i = 0; i < size(); ++i)
        intervals.push_back({std::numeric_limits<float>::max(), 0});

    // 初始化
    for (auto &&data : m_vData)
    {
        data.color = VData::Color::White;
        data.data = 0;
    }

    std::size_t count = 0;
    std::stack<std::stack<std::size_t>> sts;
    sts.push(visit(s));

    while (!sts.empty())
    {
        while (!sts.top().empty())
        {
            auto u = sts.top().top();
            sts.top().pop();

            std::get<0>(intervals.at(u)) = std::min(std::get<0>(intervals.at(u)), m_vData.at(u).data);
            if (m_vData.at(u).color == VData::Color::Black)
            {
                count = m_vData.at(u).data;
                std::get<1>(intervals.at(u)) = count;
            }
            else
            {
                m_vData.at(u).data = count;
                sts.push(std::move(visit(u)));
            }
        }
        sts.pop();
    }

    return intervals;
}

std::stack<std::size_t> Graph_Sparse::visit(std::size_t u)
{
    auto checkIt = [&](std::tuple<std::size_t, List<std::size_t>::Iterator> pair) {
        // 检查 it 是否到达 u 子节点的尾部
        return m_adjList.at(std::get<0>(pair))->end() != std::get<1>(pair);
    };

    auto checkColor = [&](std::size_t v) { return m_vData.at(v).color == VData::Color::White; };

    std::stack<std::size_t> st;
    std::size_t count = m_vData.at(u).data;

    auto it = m_adjList.at(u)->begin();
    st.push(u);
    m_vData.at(u).color = VData::Color::Gray;
    m_vData.at(u).data = ++count;

    while (checkIt({u, it}))
    {
        if (checkColor(*it))
        {
            u = *it;
            it = m_adjList.at(u)->begin();
            st.push(u);
            m_vData.at(u).color = VData::Color::Gray;
            m_vData.at(u).data = ++count;
        }
        else
            ++it;
    }

    // 开始返回
    m_vData.at(u).color = VData::Color::Black;
    return st;
}
```

利用 `std::function` 可以实现 lambda 函数递归，效率更高

```cpp
std::vector<std::tuple<float, float>> Graph_Sparse::dfs_recursive(std::size_t s)
{
    std::vector<std::tuple<float, float>> intervals(size());

    // 初始化
    for (auto &&data : m_vData)
    {
        data.color = VData::Color::White;
        data.data = 0;
    }

    std::size_t count = 0;
    std::function<void(std::size_t u)> dfs_visit = [&](std::size_t u) {
        count++;
        intervals.at(u) = {count, 0};
        m_vData.at(u).color = VData::Color::Gray;
        for (auto it = m_adjList.at(u)->begin(); it != m_adjList.at(u)->end(); ++it)
            if (m_vData.at(*it).color == VData::Color::White)
                dfs_visit(*it);
        m_vData.at(u).color = VData::Color::Black;
        count++;
        intervals.at(u) = {std::get<0>(intervals.at(u)), count};
    };
    
    dfs_visit(s);
    return intervals;
}
```



## 对象

### 初始化

#### 列表初始化

建议使用 `{}` 进行初始化，区别于仿函数

```cpp
struct A {
    int x;
    int y;

    int operator()() const {
        return x + y;
    }
};

printf("%d\n", A{1, 2}());
```

另外，这种写法也利于编译器正确识别参数，其限制更严格。例如

```cpp
int(3.14f)	// 通过
int{3.14f}	// 报错
```

从而避免精度丢失。


#### 多参数初始化

对于具有多个参数的构造类，最好将所有数据打包为单独的结构体，这样初始化时可以只传入结构体

```cpp
struct Data {
    int age;
    std::string name;
    std::string address;
    int score;
    int telephone;
};

class A {
  private:
    Data data;

  public:
    // 容易出错，尤其在需要频繁调整 Data 结构时
    A(int age, const std::string& name, const std::string& address, int score = 80, int telephone = 110): data{age, name, address, score, telephone} {}

    // 不易出错
    A(Data data): data(data) {}
};
```

多参数的构造函数调用，需要依次指定每个参数，如果存在多个默认值，可能为了调整某个默认值，不得不写出其它默认值

```cpp
// 想修改 telephone，不得不写出 score
A a(20, "Alice", "Beijing", 80, 120);
```

借助 C++20 的初始化方法，可以很容易

```cpp
A b(Data{.age = 20, .name = "Alice", .address = "Beijing", .telephone = 123456789});
```



#### 外部构造器

对于多参数，最好将构造与数据分离

```cpp
struct Data {
    int age;
    std::string name;
    std::string address;
    int score;
    int telephone;

	// 通过参数结构构造
    A construct(int fd) {
        A a(fd);
        a.data = *this;
        return a;
    }
};

class A {
    friend class Data;

  private:
    int fd;
    Data data;

  public:
    A(int fd): fd(fd) {}
    A(const A&) = default;
};

A a = Data{.age = 20, .name = "Alice", .address = "Beijing", .score = 90, .telephone = 123456789}.construct(10);
```



#### 外部移动构造

进一步提供单参数构造，利用返回引用来重复构造新的参数；并且使用移动语义来提高效率

```cpp
class A;

struct Data {
    int age;
    std::string name;
    std::string address;
    int score;
    int telephone;

    [[nodiscard]] A construct(int fd) &&;

    [[nodiscard]] Data&& withAge(int age) && {
        this->age = age;
        return std::move(*this);
    }

    [[nodiscard]] Data&& withName(std::string name) && {
        this->name = name;
        return std::move(*this);
    }

    [[nodiscard]] Data&& withAddress(std::string address) && {
        this->address = address;
        return std::move(*this);
    }
};

class A {
    friend class Data;

  private:
    int fd;
    Data data;

  public:
    A(int fd): fd(fd) {}
    A(A&&) = default;
    A(const A&) = delete;
};

A Data::construct(int fd) && {
    A a(fd);
    a.data = *this;
    return a;
}

A a = Data{}.withAddress("Shanghai").withAge(20).withName("Bob").construct(100);
```

这种构造方法的好处在于能够在构造某个参数时，在函数内部提供对其它参数的调整构造。



#### 外部强制构造

通过使用模板参数，强制某个参数必须构造。例如

```cpp
class A;

struct DataBase {
    int age;
    std::string name;
    std::string address;
    int score;
    int telephone;
};

template <bool Ready = false> struct Data : DataBase {
    A construct(int fd);

    Data& withAge(int age) {
        this->age = age;
        return *this;
    }

	// 只有当调用了 withName 后，转换为 Data<true>
    Data<true>& withName(std::string name) {
        this->name = name;
        // 利用基类引用转换来间接转换，否则不能直接转换为 Data<true>&
        return static_cast<Data<true>&>(static_cast<DataBase&>(*this));
    }

    Data& withAddress(std::string address) {
        this->address = address;
        return *this;
    }
};

class A {
    friend class Data<true>;

  private:
    int fd;
    Data<true> data;

  public:
    A(int fd): fd(fd) {}
    A(const A&) = default;
};

template <bool Ready> A Data<Ready>::construct(int fd) {
	// 要求 Ready == true
    static_assert(Ready, "Data is not ready yet");
    A a(fd);
    a.data = *this;
    return a;
}

A a = Data<false>{}.withAddress("Shanghai").withAge(20).withName("Bob").construct(100);	// 通过
A b = Data<false>{}.withAddress("Shanghai").withAge(20).construct(100);					// 报错
```



#### 静态构造

对于特殊的类型构造，例如对于矩阵类构造单位矩阵，就可以使用静态构造，这样能够与一般的构造相区分

```cpp
class Matrix {
private:
    int rows;
    int cols;
    std::vector<std::vector<int>> data;

public:
	Matrix(int rows, int cols): rows(rows), cols(cols), data(rows, std::vector<int>(cols)) {}
    
    static Matrix identity(int n) {
        Matrix a(n, n);
        for(int i = 0; i < n; i++) {
            a.data[i][i] = 1;
        }
        return a;
    }
};
```



### 局部对象

如果需要在源文件中定义临时使用的对象，应当使用匿名命名空间

```cpp
namespace {

	struct A {
	    int x;
	    int y;
	
	    int operator()() const {
	        return x + y;
	    }
	};

}
```

包含在 `namespace` 中的对象只在文件内部可见，因此避免了多个源文件中定义同名对象的冲突。



### 成员函数

#### Explicit

为了避免单参数构造函数出现的隐式转换, 应当使用显式构造

```cpp
struct A {
    int x;
    int y;

    explicit A(int x): x(x), y(x * x) {}
};

A a = 10;	// 报错
A b(10);	// 显式调用
```

另外，将构造函数声明为 `explicit` 可以检测隐式的拷贝构造

```cpp
// 需要拷贝构造传入
void fun(A a) {}

A a(10);
fun(a);	// 报错
```



#### Set/Get

只有当一个成员修改后，需要同步修改其它成员时，才需要指定 `Set/Get` 成员，否则不仅更麻烦，还会有性能损失。

> [!note] 
> 一个类的所有公共方法和属性都必须确保对它们的任何操作不会导致非法情况出现。



#### 拷贝赋值

拷贝赋值函数可以返回自身的引用，这样可以支持连等号赋值

```cpp
struct A {
    int x;
    int y;

    explicit A(int x): x(x), y(x * x) {}

    A& operator=(const A& other) {
        x = other.x;
        y = other.y;
        return *this;
    }
};
```



#### 防止自赋值

在拷贝赋值和移动赋值函数中，需要检查是否是自赋值。由于自赋值的情况很少，可以使用 `[[likely]]` 来提供优化

```cpp
struct A {
    A& operator=(A&& other) {
        if(this != &other) [[likely]] {
            // 移动
        }
        return *this;
    }
};
```



#### 定义规则

前人总结出关于拷贝、移动构造和赋值的基本定义规则

1. 如果定义了析构函数，必须定义或删除拷贝构造和拷贝赋值（防止指针浅拷贝导致释放两次）
2. 如果定义了拷贝构造，必须定义或删除拷贝赋值
3. 如果定义了移动构造，必须定义或删除移动赋值
4. 如果定义了拷贝构造或拷贝赋值，必须定义移动构造或移动赋值



### 多接口

#### 公共接口

考虑下面的代码，Food 类提供了两个接口，分别被多个子类重写。这种写法的问题在于，如果需要添加新的接口，就需要修改基类和子类

```cpp
struct EatParams {
    int amount;
    int speed;
};

struct DrinkParams {
    int amount;
    int temperature;
};

struct Food {
    virtual void eat(const EatParams& params) const = 0;
    virtual void drink(const DrinkParams& params) const = 0;
};

struct Cake : Food {
    void eat(const EatParams& params) const override { std::cout << "Eating a cake with " << params.amount << " grams of flour and " << params.speed << " speed." << std::endl; }
};

struct Milk : Food {
    void drink(const DrinkParams& params) const override { std::cout << "Drinking " << params.amount << " milliliters of milk at " << params.temperature << " degrees Celsius." << std::endl; }
};

struct Pudding : Food {
    void eat(const EatParams& params) const override { std::cout << "Eating a pudding with " << params.amount << " grams of sugar and " << params.speed << " speed." << std::endl; }

    void drink(const DrinkParams& params) const override { std::cout << "Drinking " << params.amount << " milliliters of water at " << params.temperature << " degrees Celsius." << std::endl; }
};
```



#### 分离接口

一种改进方案是将接口分离为两个，然后子类分别去继承这些接口

> 虽然 Pudding 继承了两个接口，不过这两个结构都是空的，因此不会有问题。

```cpp
struct Eatable {
    virtual void eat(const EatParams& params) const = 0;
};

struct Drinkable {
    virtual void drink(const DrinkParams& params) const = 0;
};

struct Cake : Eatable {
    void eat(const EatParams& params) const override { std::cout << "Eating a cake with " << params.amount << " grams of flour and " << params.speed << " speed." << std::endl; }
};

struct Milk : Drinkable {
    void drink(const DrinkParams& params) const override { std::cout << "Drinking " << params.amount << " milliliters of milk at " << params.temperature << " degrees Celsius." << std::endl; }
};

struct Pudding : Eatable, Drinkable {
    void eat(const EatParams& params) const override { std::cout << "Eating a pudding with " << params.amount << " grams of sugar and " << params.speed << " speed." << std::endl; }

    void drink(const DrinkParams& params) const override { std::cout << "Drinking " << params.amount << " milliliters of water at " << params.temperature << " degrees Celsius." << std::endl; }
};
```

但这种写法在应用多态时会比较麻烦

```cpp
void run(const Eatable& eatable) {
	EatParams params{100, 10};
	eatable->eat(params);
}

void run(const Drinkable& drinkable) {
	DrinkParams params{100, 10};
	drinkable->drink(params);
}

int main() {
    Cake cake;
    Milk milk;
    Pudding pudding;

	// 需要转换
    run(static_cast<Eatable&>(cake));
    run(static_cast<Drinkable&>(milk));
    run(static_cast<Eatable&>(pudding));
    run(static_cast<Drinkable&>(pudding));
    return 0;
}
```



#### 虚继承

再抽象出一个接口基类，接口继承应用虚继承，防止重复继承接口。然后利用动态转换统一调用

```cpp
struct Food {
    virtual ~Food() = default;
};

struct Eatable : virtual Food {
    virtual void eat(const EatParams& params) const = 0;
};

struct Drinkable : virtual Food {
    virtual void drink(const DrinkParams& params) const = 0;
};

void run(Food* food) {
    // 新语法
    if(auto eat = dynamic_cast<Eatable*>(food)) {
        EatParams params{100, 10};
        eat->eat(params);
    }

    if(auto drink = dynamic_cast<Drinkable*>(food)) {
        DrinkParams params{500, 20};
        drink->drink(params);
    }
}

int main() {
    Cake cake;
    Milk milk;
    Pudding pudding;

    run(&cake);
    run(&milk);
    run(&pudding);
    run(&pudding);
    return 0;
}
```



#### 代替动态转换

在不使用动态转换的情况下，可以定义转换接口来实现相同的效果

```cpp
struct Eatable;
struct Drinkable;

struct Food {
    virtual ~Food() = default;

    virtual Drinkable* toDrinkable() { return nullptr; }
    virtual Eatable* toEatable() { return nullptr; }
};

struct Eatable : virtual Food {
    virtual void eat(const EatParams& params) const = 0;

    Eatable* toEatable() override { return this; }
};

struct Drinkable : virtual Food {
    virtual void drink(const DrinkParams& params) const = 0;

    Drinkable* toDrinkable() override { return this; }
};

void run(Food* food) {
    if(auto eat = food->toEatable()) {
        EatParams params{100, 10};
        eat->eat(params);
    }

    if(auto drink = food->toDrinkable()) {
        DrinkParams params{500, 20};
        drink->drink(params);
    }
}
```

但这种写法需要前向声明，并且每次创建一个新的接口，就需要提供多个转换接口，非常麻烦。



#### 访问者模式

为了统一类的行为，可以抽象出 `FoodVisitor` 接口，对每个功能提供一个 `visit` 访问接口；然后每个食物类提供一个 `accept` 接受访问，只需要重复相同的访问代码即可，便于扩展。

```cpp
struct Eatable;
struct Drinkable;

struct FoodVisitor {
    virtual void visit(Eatable* eatable){};
    virtual void visit(Drinkable* drinkable){};
    virtual ~FoodVisitor() = default;
};

struct Food {
    virtual void accept(FoodVisitor* visitor) = 0;
    virtual ~Food() = default;
};

struct Eatable : virtual Food {
    virtual void eat(const EatParams& params) const = 0;

    void accept(FoodVisitor* visitor) override { visitor->visit(this); }
};

struct Drinkable : virtual Food {
    virtual void drink(const DrinkParams& params) const = 0;

    void accept(FoodVisitor* visitor) override { visitor->visit(this); }
};
```

对于同时继承多个食物的类，让访问器被两种接口接收

```cpp
struct Pudding : Eatable, Drinkable {
	// ...

	// 访问两种属性
    void accept(FoodVisitor* visitor) override {
        Eatable::accept(visitor);
        Drinkable::accept(visitor);
    }
};
```

最后继承出具体的访问行为

```cpp
struct FoodRunner : FoodVisitor {
    void visit(Eatable* eatable) override { eatable->eat(EatParams{100, 10}); }
    void visit(Drinkable* drinkable) override { drinkable->drink(DrinkParams{500, 20}); }
};

int main() {
	FoodRunner runner;
    
    Cake cake;
    Milk milk;
    Pudding pudding;

    cake.accept(&runner);
    milk.accept(&runner);
    pudding.accept(&runner);

    return 0;
}
```



### 移动语义

#### 删除拷贝移动

提高对象转移效率的最基本方法是删除所有拷贝和移动构造、赋值函数，然后使用智能指针包装

```cpp
struct Resource {
    Resource() { std::cout << "Resource acquired" << std::endl; }
	~Resource() { std::cout << "Resource released" << std::endl; }

    // 只要删除移动构造函数，会同时删除拷贝构造函数和赋值运算符
    Resource(Resource&& other) = delete;

	int* p;
};

void func(std::shared_ptr<Resource> resource) {
}

auto x = std::make_shared<Resource>();
func(x);
```

> [!note] 
> 建议所有资源类都删除拷贝和移动操作，统一使用智能指针。



#### 封装资源

可以将资源封装为内部结构，保存为智能指针，提供外部访问的重载运算符

```cpp
struct Resource {
    // 内部资源
    struct Self {
        int* value;

        Self(): value(new int(42)) { std::cout << "Self acquired" << std::endl; }

        // 内部方法
        void speak() { std::cout << "Hello, world!" << std::endl; }

        ~Self() {
            delete value;
            std::cout << "Self released" << std::endl;
        }
    };

  private:
    std::shared_ptr<Self> self;

  public:
    Resource(): self(std::make_shared<Self>()) {}
    Resource(std::shared_ptr<Self> s): self(s){};

    // 支持深拷贝
    Resource clone() const { return std::make_shared<Self>(*self); }

    // 访问内部资源
    Self* operator->() { return self.get(); }
};

void func(Resource x) {
    x->speak();
}
```

或者可以将内部方法实现在外部类中。

#### 增加功能

利用 `requires` 来判断是否存在相应的成员函数来提供调用；并且可以添加新的功能，例如克隆出智能指针

```cpp
struct MsgBase {
    virtual void speak() = 0;
    virtual void happy() = 0;
    virtual std::shared_ptr<MsgBase> clone() const = 0;
    virtual ~MsgBase() = default;
};

template <class Msg> struct MsgImpl : MsgBase {
    Msg msg;

    template <typename... Args> MsgImpl(Args&&... args): msg(std::forward<Args>(args)...) {}

    void speak() override { msg.speak(); }

    void happy() override {
        if constexpr(requires { msg.happy(); }) {
            msg.happy();
        } else {
            std::cout << "No happy message available" << std::endl;
        }
    }

    std::shared_ptr<MsgBase> clone() const override {
        return std::make_shared<MsgImpl>(msg);
    }
};
```

## 智能指针

### 取代指针

使用 `unique_ptr` 代替一般的指针构造临时对象

```cpp
// 使用 std::make_unique 代替 new
auto c = std::make_unique<A>();
```



### 自动释放

利用 `unique_ptr` 自动管理内存的特性，实现 RAII 管理局部指针。例如打开文件时

```cpp
auto file = std::unique_ptr<FILE>(std::fopen("test.txt", "a+"));
```

这样当离开作用域时，指针会自动释放。不过文件不会自动关闭，因此还需要指定删除器

```cpp
using del = void (*)(FILE*);
auto file = std::unique_ptr<FILE, del>(std::fopen("test.txt", "a+"), [](FILE* f) {
    if(f != nullptr) {
        std::fclose(f);
        delete f;
    }
});
```



### 转移指针

可以将 `unique_ptr` 隐式转换为 `shared_ptr`，因此需要返回指针的函数都可以选择返回 `unique_ptr` 类型。对于需要裸指针所有权的古老函数，可以使用 `release()` 转移指针所有权。



### 类型擦除

使用 `unique_ptr` 创建的指针在删除时默认会按照模板类型删除，这就导致如果将子类指针保存在基类 `unique_ptr` 中，有可能发生内存泄漏。而使用 `shared_ptr` 创建的指针则不会有内存泄漏，因为它会对 deleter 做类型擦除，在创建时就会保存指针类型。



### 使用场景

根据实际需要判断使用哪一种智能指针

1. `unique_ptr` 如果一个对象仅属于我时使用，例如父窗口拥有子窗口的指针
2. 原始指针：当我指向的对象被释放时，我也被释放时使用，例如子窗口指向父窗口
3. `shared_ptr` 当需要多个对象共享，或者虽然一个对象仅属于我，但是需要使用 `weak_ptr` 时使用
4. `weak_ptr` 当一个对象不属于我，并且它释放时我不一定释放时使用，例如指向窗口上一次被点击的元素



### 异常处理

当抛出异常时，需要注意内存释放问题。例如 `throw` 位置，所有申请内存的局部变量最好都能够自动释放内存，例如使用 `unique_ptr` 构建。

> [!note]
> 异常处理应当与 RAII 相结合，即使抛出异常，析构函数调用仍能确保资源被顺利释放。



## 多线程安全

### 线程控制

#### jthread

建议使用 `std::jthread` 作为线程，它会在析构时自动 `join`，等待线程执行完毕。

```cpp
#include <iostream>

void print_hello() {

}

void print_world() {

    a = "World!";
    std::cout << a << std::endl;
}

int main() {
    std::vector<std::jthread> threads;
    threads.push_back(std::jthread(print_hello));
    threads.push_back(std::jthread(print_world));
    threads.clear();
    return 0;

```



#### 线程顺序

假如我们希望一个线程在另一个线程执行之前阻塞，就可以利用循环锁定。然而，由于编译器优化的问题，最好不要使用空转的循环，例如

```cpp
while(a == "") ;
```

改为使用 `std::unique_lock` 在循环内解锁，允许其它线程访问；同时手动 `yield` 切换到其它活跃线程，之后再重新锁定。下面代码控制 `a` 必须先赋值为 `Hello!` 然后才赋值为 `World!`，实现顺序控制。

```cpp
#include <iostream>
#include <mutex>
#include <thread>
#include <vector>

std::mutex mtx;
std::string a;

void print_hello() {
    // sleep 一定要放在锁外边，不然浪费锁定
    std::this_thread::sleep_for(std::chrono::milliseconds(5000));
    std::unique_lock lock(mtx);
    a = "Hello!";
    std::cout << a << std::endl;
}

void print_world() {
    // 使用 unique_lock 可以手动解锁
    std::unique_lock lock(mtx);
    while(a == "") {
        // 暂时释放锁，切换到其它活跃线程，这样有机会修改 a
        lock.unlock();
        std::this_thread::yield();

        // 重新锁定，防止其它线程修改 a
        lock.lock();
    }
    a = "World!";
    std::cout << a << std::endl;
}

int main() {
    std::vector<std::jthread> threads;
    threads.push_back(std::jthread(print_world));
    threads.push_back(std::jthread(print_hello));
    return 0;
}
```



#### 访问者模式

将数据封装在内部结构体中，利用 RAII 一次性上锁。

```cpp

class MTVector {
  std::vector<double> m_arr;
  std::mutex m_mtx;

public:
  // 访问者
  class Accessor {
    MTVector &m_that;
    std::unique_lock<std::mutex> m_guard;

  public:
    // 自动上锁
    Accessor(MTVector &that) : m_that(that), m_guard(that.m_mtx) {}

    void push_back(int val) const { return m_that.m_arr.push_back(val); }

    size_t size() const { return m_that.m_arr.size(); }
  };

  // 利用访问者构造上锁
  Accessor access() { return {*this}; }
};

void test() {
  MTVector arr;

  std::thread t1([&]() {
    auto axr = arr.access();
    for (int i = 0; i < 1000; i++)
      axr.push_back(i);
  });

  std::thread t2([&]() {
    auto axr = arr.access();
    for (int i = 0; i < 1000; i++)
      axr.push_back(i);
  });
}
```



### 条件变量

上述循环体虽然避免了编译器错误优化，但是当另一个线程 sleep 时会造成循环空转。解决方案就是使用条件变量，它会在不满足条件时挂起线程，避免了 CPU 空转。

```cpp
std::condition_variable cv;
std::mutex mtx;
std::string a;

void print_hello() {
    // sleep 一定要放在锁外边，不然浪费锁定
    std::this_thread::sleep_for(std::chrono::milliseconds(5000));
    std::unique_lock lock(mtx);
    a = "Hello!";
    std::cout << a << std::endl;

    // 通知另一个线程状态更新
    cv.notify_one();
}

void print_world() {
    // 使用 unique_lock 可以手动解锁
    std::unique_lock lock(mtx);
    while(a == "") {
        // 等待状态更新
        cv.wait(lock);
    }
    a = "World!";
    std::cout << a << std::endl;
}
```



#### wait

虽然 `wait` 会阻塞线程，但是依然有必要使用 `while`，因为有可能出现另一个线程很快执行完毕，发送消息时这边线程还没有开始，导致一直阻塞的问题。例如

```cpp
void print_world() {
	// 等待 10 秒，这样另一个线程很快执行完毕，发送通知时这边还没有 wait
	std::this_thread::sleep_for(std::chrono::milliseconds(10000));

    // 使用 unique_lock 可以手动解锁
    std::unique_lock lock(mtx);

	// 一直阻塞
	cv.wait(lock);
	
    a = "World!";
    std::cout << a << std::endl;
}
```

而添加循环后相当于多一层保险：如果另一个线程已经执行完，那么 `a` 已经非空，不会进入内部阻塞；如果还未执行完，才会进入内部阻塞。



可以将循环封装为 lambda 表达式，作为退出 `wait` 的条件，更加直观

```cpp
// 注意这里是退出条件，不是循环条件
cv.wait(lock, [] { return a != ""; });
```

在 `wait` 过程中会解锁，以便其它线程可以调用 `cv` 发送消息。

> [!note]
>
> 之所以要将 `wait, unique_lock` 共用，是因为要确保多个线程被唤醒时，只有一个能够被启动（`notify_one`）。



#### wait_for

可以指定等待时间，如果超时则会返回等待结果

```cpp
using namespace std::literals;

auto res = cv.wait_for(lock, 1s, [] { return a != ""; });
if(res) {
    std::cout << "success" << std::endl;
} else {
    std::cout << "timeout" << std::endl;
}
```



### 消息队列

#### 封装消息处理

使用互斥量配合条件变量，实现一个基本的多线程消息队列处理

```cpp
std::deque<std::string> a;

void print_hello() {
    {
        std::unique_lock lock(mtx);
        a.push_back("Hello!");
        cv.notify_one();
    }
    std::this_thread::sleep_for(1s);
    {
        std::unique_lock lock(mtx);
        a.push_back("World!");
        cv.notify_one();
    }
    std::this_thread::sleep_for(1s);
    {
        std::unique_lock lock(mtx);
        a.push_back("Exit");
        cv.notify_one();
    }
}

void print_world() {
    while(true) {
        std::unique_lock lock(mtx);
        cv.wait(lock, [&] { return !a.empty(); });
        std::cout << "message: " << a.back() << std::endl;
        if(a.back() == "Exit") {
            break;
        }
        a.pop_back();
    }
}
```

将这一过程封装为一个消息队列

```cpp
#include <condition_variable>
#include <deque>
#include <mutex>
#include <optional>
#include <thread>

template <class T> class mt_queue {
  private:
    std::deque<T> m_queue;
    std::mutex m_mutex;
    std::condition_variable m_cv;

  public:
    void push(T const& value) {
        std::unique_lock lock(m_mutex);
        m_queue.push_front(value);
        m_cv.notify_one();
    }

    T pop() {
        std::unique_lock lock(m_mutex);
        m_cv.wait(lock, [this] { return !m_queue.empty(); });
        auto value = std::move(m_queue.back());
        m_queue.pop_back();
        return value;
    }

    std::optional<T> try_pop() {
        std::unique_lock lock(m_mutex);
        if(m_queue.empty()) {
            return std::nullopt;
            auto value = std::move(m_queue.back());
            m_queue.pop_back();
            return value;
        }
    }

    std::optional<T> try_pop_for(std::chrono::milliseconds timeout) {
        std::unique_lock lock(m_mutex);
        if(m_cv.wait_for(lock, timeout, [this] { return !m_queue.empty(); })) {
            auto value = std::move(m_queue.back());
            m_queue.pop_back();
            return value;
        }
        return std::nullopt;
    }
};
```



#### 隐藏锁定

使用封装的消息队列，可以将多线程的变量操作隐藏起来，避免了互斥锁、条件变量的显式使用。例如原本可以通过原子变量实现多线程求和运算

```cpp
#include <iostream>
#include <thread>
#include <vector>

std::atomic_int counter = 0;

void compute_atomic(int beg, int end) {
    for(int i = beg; i < end; i++) {
        counter += i;
    }
}

int main() {
    std::vector<std::jthread> threads;

    // 分成 10 个线程计算
    for(int i = 0; i < 10000; i += 1000) {
        threads.push_back(std::jthread(compute, i, i + 1000));
    }
    threads.clear();
	std::cout << "Counter: " << counter << std::endl;    
    return 0;
}
```

可以拆分为使用队列进行处理，并且由于队列可以保存值，因此避免了全局变量的使用，在 `compute` 中只推入数据，而在 `comput_thread` 中取出数据并进行计算

```cpp
#include <iostream>
#include <thread>
#include <vector>
#include "mtqueue.hpp"

mt_queue<std::optional<int>> counter_queue;

void compute_thread() {
    int counter = 0;
    int count_complte = 0;
    while(count_complte < 10) {
        auto i = counter_queue.pop();

        // 遇到计数完成信号
        if(i == std::nullopt) {
            count_complte++;
            continue;
        }
        counter += i.value();
    }
    std::cout << "Counter: " << counter << std::endl;
}

void compute(int beg, int end) {
    for(int i = beg; i < end; i++) {
        counter_queue.push(i);
    }
    // 推入计数完成信号
    counter_queue.push(std::nullopt);
}

int main() {
    std::vector<std::jthread> threads;

    // 分成 10 个线程计算
    for(int i = 0; i < 10000; i += 1000) {
        threads.push_back(std::jthread(compute, i, i + 1000));
    }
    // 最终计算线程
    threads.push_back(std::jthread(compute_thread));
    threads.clear();
    return 0;
}
```



#### 模拟屏障

在 C++20 中引入了 `std::barrier` 用于同步线程

```cpp
#include <barrier>
#include <iostream>
#include <thread>
#include <vector>

#include "mtqueue.hpp"

// 设置 11 个线程的屏障
std::barrier barrier(11);
mt_queue<std::optional<int>> counter_queue;

void compute_thread() {
    int counter = 0;
    while(true) {
        auto i = counter_queue.pop();

        // 遇到计数完成信号
        if(i == std::nullopt) {
            break;
        }
        counter += i.value();
    }
    std::cout << "Counter: " << counter << std::endl;
}

void compute(int beg, int end) {
    for(int i = beg; i < end; i++) {
        counter_queue.push(i);
    }
    std::cout << "Compute thread " << beg << " to " << end << " begin" << std::endl;
    // barrier.arrive_and_drop();	// barrier 到达，当本线程执行完成后，立即退出
    barrier.arrive_and_wait();		// barrier 到达，等待直到 barrier 解除
    std::cout << "Compute barrier " << beg << " to " << end << " done" << std::endl;
}

void finish_thread() {
    std::cout << "Finish thread" << std::endl;
    barrier.arrive_and_wait();
    std::cout << "Finish barrier" << std::endl;
    counter_queue.push(std::nullopt);
}

int main() {
    std::vector<std::jthread> threads;

    // 分成 10 个线程计算
    for(int i = 0; i < 10000; i += 1000) {
        threads.push_back(std::jthread(compute, i, i + 1000));
    }
    // 最终计算线程
    threads.push_back(std::jthread(compute_thread));
    threads.push_back(std::jthread(finish_thread));
    threads.clear();
    return 0;
}
```

#### 任务队列

可以利用函数包装器来保存任务队列，以状态参数的引用作为参数，每个状态只需要单独定义在每个线程中即可，避免了全局变量。

```cpp
#include <barrier>
#include <functional>
#include <iostream>
#include <sstream>
#include <thread>
#include <vector>

#include "mtqueue.hpp"

using namespace std::literals;

struct CounterState {
    int count = 0;
    std::string msg = "";
};

// 任务队列
mt_queue<std::function<void(CounterState&)>> counter_queue(1000);

void compute_thread() {
    CounterState state;
    // 每次最多等待 100ms 获得任务，没获得任务则退出循环
    while(auto task = counter_queue.try_pop_for(100ms)) {
        auto f = task.value();
        f(state);
    }

    // 暂时输出到字符串流，最后一次性输出到控制台，防止输出混乱
    std::ostringstream oss;
    oss << "msg: " << state.msg << std::endl;
    oss << "count: " << state.count << std::endl;
    std::cout << oss.str();
}

void compute(int beg, int end) {
    for(int i = beg; i < end; i++) {
        // 按值捕获，防止数据竞争
        counter_queue.push([i2 = i + 1](CounterState& state) { state.count += i2; });
    }
    counter_queue.push([](CounterState& state) { state.msg = "Done"; });
}

int main() {
    std::vector<std::jthread> threads;

    // 分成 10 个线程计算
    for(int i = 0; i < 10000; i += 1000) {
        threads.push_back(std::jthread(compute, i, i + 1000));
    }
    // 分成 3 个线程处理
    threads.push_back(std::jthread(compute_thread));
    threads.push_back(std::jthread(compute_thread));
    threads.push_back(std::jthread(compute_thread));
    threads.clear();
    return 0;
}
```

#### 暂停队列

添加一个 `finish` 成员来控制线程执行状态

```cpp
#include <barrier>
#include <functional>
#include <iostream>
#include <sstream>
#include <thread>
#include <vector>

    }
    counter_queue.push([](CounterState& state) { state.msg = "Done"; });
}

int main() {
    std::vector<std::jthread> compute_pool;
    std::vector<std::jthread> counter_pool;

    // 推入计算任务后，推入结束信号
    counter_queue.push([](CounterState& state) { state.finished = true; });
    counter_queue.push([](CounterState& state) { state.finished = true; });
    counter_queue.push([](CounterState& state) { state.finished = true; });

    // 分成 3 个线程处理所有任务
    counter_pool.push_back(std::jthread(compute_thread));
```

#### 应用场景

在需要并发的场景，建议使用消息队列处理。这种场景的主要目的不在于追求极致的性能，而是尽可能同步处理多个消息来源，例如网络服务端对客户端消息的并发处理。



### 原子操作

#### 原子性

一个原子量的成员函数通常都是原子的，但是如果将它们分开使用，就会丧失原子性

```cpp
std::atomic<int> x;

x = x + 1;  // 非原子

x.store(x.load() + 1);  // 非原子
```



#### volatile

被 `volatile` 修饰的变量不会被编译器优化

```cpp
#include <iostream>

    volatile int flag = 0;
    
    while (flag == 0) {

    }

    std::cout << "Flag is now non-zero.\n";

```

在多线程时可用于保证变量处理的正确性。然而，这种写法只在嵌入式领域使用，在 C++ 标准中没有规定 volatile 在多线程操作时的正确性。标准的做法还是使用原子变量。

### 内存管理

#### 内存监视

可以通过重载 `new/delete` 运算符实现内存监视

```cpp
// 注意 windows.h 要放在前面
#include <windows.h>
//
#include <dbghelp.h>
#include <intrin.h>

#include <iostream>
#include <map>

#pragma comment(lib, "dbghelp.lib")

// 获得当前函数上一层的上一层的函数名
std::string get_allocate_function() {
    std::string name;

    // Initialize DbgHelp library
    HANDLE process = GetCurrentProcess();
    SymInitialize(process, NULL, TRUE);

    // Capture the stack trace
    void* stack[3];
    USHORT frames = CaptureStackBackTrace(0, 3, stack, NULL);

    if(frames < 3) {
        std::cout << "Unable to capture stack trace." << std::endl;
        SymCleanup(process);
        return name;
    }

    // Allocate memory for the symbol information
    SYMBOL_INFO* symbol = (SYMBOL_INFO*)calloc(sizeof(SYMBOL_INFO) + 256 * sizeof(char), 1);
    symbol->MaxNameLen = 255;
    symbol->SizeOfStruct = sizeof(SYMBOL_INFO);

    // The caller's address is in stack[1]
    DWORD64 caller_address = (DWORD64)(stack[2]);
    DWORD64 displacement = 0;

    // Attempt to get the symbol from the address
    if(SymFromAddr(process, caller_address, &displacement, symbol)) {
        name = symbol->Name;
    }

    // Free allocated memory
    free(symbol);

    // Cleanup DbgHelp library
    SymCleanup(process);

    return name;
}

struct AllocatedInfo {
    void* ptr;
    size_t size;
    std::string caller;
};

struct GlobalDebuger {
    // 注意不能在这个位置初始化，会导致编译器报错
    bool enable;
    std::map<void*, AllocatedInfo> allocated;

    // 不要使用列表初始化 enable，会导致编译器报错
    GlobalDebuger() {
        enable = true;
        std::cout << "GlobalDebuger initialized." << std::endl;
    }

    ~GlobalDebuger() {
        enable = false;
        std::cout << "GlobalDebuger destroyed." << std::endl;
        for(auto& [ptr, info]: allocated) {
            printf("内存泄漏：ptr = %p, size = %zu, caller = %s\n", ptr, info.size, info.caller.c_str());
        }
    }
} global_debuger;

struct EnableGuard {
    bool was_enabled;

    EnableGuard() {
        // 记录锁定之前的状态
        was_enabled = global_debuger.enable;
        global_debuger.enable = false;
    }

    ~EnableGuard() { global_debuger.enable = was_enabled; }

    operator bool() const { return was_enabled; }
};

void* operator new(size_t size) {
    EnableGuard guard;
    void* ptr = malloc(size);

    // 取消 enable 标志，避免 map 创建元素时 new 递归调用
    if(guard && ptr) {
        std::string caller = get_allocate_function();
        printf("new: ptr = %p, size = %zu, caller = %s\n", ptr, size, caller.c_str());
        global_debuger.allocated[ptr] = AllocatedInfo{ptr, size, caller};
    }
    if(ptr == nullptr) throw std::bad_alloc();
    return ptr;
}

void operator delete(void* ptr) noexcept {
    EnableGuard guard;
    if(guard && ptr) {
        printf("delete: ptr = %p, caller = %s\n", ptr, global_debuger.allocated[ptr].caller.c_str());
        global_debuger.allocated.erase(ptr);
    }
    free(ptr);
}

void test() {
    int* p = new int(10);
    short* s = new short;
    delete p;
}

int main() {
    test();
    return 0;
}
```

## 设计模式

### 基本思路

在[[设计模式]]部分我们已经给出两个重要的代码设计思路

- 开放封闭原则：尽可能不修改源码，而是添加新代码来实现新功能
- 依赖倒置原则：对每个功能抽象出接口，便于解耦和



### 策略模式

策略模式的思路是将不同的策略提取出来单独封装，便于事后添加新的策略。例如有两个函数

```cpp
int sum(vector<int> v) {
    int res = 0;
    for (int i = 0; i < v.size(); i++) {
        res = res + v[i];
    }
    return res;
}

int product(vector<int> v) {
    int res = 1;
    for (int i = 0; i < v.size(); i++) {
        res = res * v[i];
    }
    return res;
}
```

我们可以将循环体中的加法策略和乘法策略抽象出接口，目标函数接收

- 一个数据容器，用于获得数据
- 一个计算策略，用于计算策略

最终得到

```cpp
// 策略
struct Reducer {
    virtual int init() = 0;
    virtual int add(int a, int b) = 0;
};

// 目标函数：一个数据容器和一个计算策略
int reduce(vector<int> v, Reducer *reducer) {
    int res = reducer.init();
    for (int i = 0; i < v.size(); i++) {
        res = reducer.add(res, v[i]);
    }
    return res;
}

// 加法策略
struct SumReducer : Reducer {
	// 提供初始状态
    int init() override {
        return 0;
    }

    int add(int a, int b) override {
        return a + b;
    }
};

// 乘法策略
struct ProductReducer : Reducer {
	// 提供初始状态
    int init() override {
        return 1;
    }

    int add(int a, int b) override {
        return a * b;
    }
};
```



### 迭代器模式

假设现在要从某个地方读取数据，作为上述策略的输入。为了解耦和，我们应该将读取过程再抽象出接口。利用迭代器模式，每次迭代进行一次读取操作，从而实现从多种不同容器中获得数据的能力。例如

```cpp
// 迭代读取器
struct Inputer {
    virtual optional<int> fetch() = 0;
};

// 目标函数
int reduce(Inputer *inputer, Reducer *reducer) {
    int res = reducer.init();
    // 循环读取数据
    while (int tmp = inputer.fetch()) {
        res = reducer.add(res, tmp);
    }
    return res;
}

// 从 cin 读取数据
struct CinInputer : Inputer {
    optional<int> fetch() override {
        int tmp;
        cin >> tmp;
        if (tmp == -1)
            return nullopt;
        return tmp;
    }
};

// 从 vector 读取数据
struct VectorInputer : Inputer {
    vector<int> v;
    int pos = 0;

    VectorInputer(vector<int> v) : v(v) {}
    
    optional<int> fetch() override {
        if (pos == v.size())
            return nullopt;
        return v[pos++];
    }
};
```

现在目标函数的输入被替换为一个读取器，根据传入的读取器来获得不同来源的数据，实现了功能分离。



### 适配器模式

前面对于 `cin` 实现读取 `-1` 则终止；对于 `vector` 实现容器为空则终止。假设我们希望为读取过程指定终止条件，就必须修改这这些迭代器的终止条件，显然我们应该进一步将这一需求抽象出来。我们指定一个终止适配器

```cpp
struct StopInputerAdapter : Inputer {
    Inputer *inputer;
    int stopMark;

	// 为读取器适配终止条件
    StopInputerAdapter(Inputer *inputer, int stopMark)
        : inputer(inputer)
        , stopMark(stopMark)
    {}

    optional<int> fetch() override {
        auto tmp = inputer.fetch();
        if (tmp == stopMark)
            return nullopt;
        return tmp;
    }
};
```

注意到适配器本身也是一个读取器，它只是为不同的读取器指定终止条件，因此可以直接作为目标函数的输入

```cpp
reduce(new StopInputerAdapter(new CinInputer(), -1), new SumReducer());      	// 从 cin 读到 -1 为止
reduce(new StopInputerAdapter(new VectorInputer(v), -1), new SumReducer());  	// 从 vector 读到 -1 为止
reduce(new VectorInputer(), new SumReducer());  								// 从 vector 读，但无需截断
```



现在我们希望提供输入的过滤器，它要能够对任何输入器进行过滤，因此应该指定一个过滤适配器。例如

```cpp
// 过滤策略接口
struct FilterStrategy {
    virtual bool shouldPass(int value) = 0;  // 返回 true 表示该值应该被丢弃
};

// 输入过滤适配器
struct FilterInputerAdapter : Inputer {
    Inputer *inputer;
    FilterStrategy *strategy;

	// 为 Inputer 指定过滤策略
    FilterInputerAdapter(Inputer *inputer, FilterStrategy *strategy)
        : inputer(inputer), strategy(strategy)
    {}

	// 
    optional<int> fetch() override {
        while (true) {
            auto tmp = inputer.fetch();
            if (!tmp.has_value()) {
                return nullopt;
            }
            if (strategy->shouldPass(tmp)) {
                return tmp;
            }
        }
    }
};
```

将新的过滤功能抽象为新的接口，并且实现不同的过滤策略，因此过滤功能可以抽象为另一个策略模式。

```cpp
// 大于阈值则过滤
struct FilterStrategyAbove : FilterStrategy {
    int threshold;

    FilterStrategyAbove(int threshold) : threshold(threshold) {}

    bool shouldPass(int value) override {
        return value > threshold;
    }
};

// 小于阈值则过滤
struct FilterStrategyBelow : FilterStrategy {
    int threshold;

    FilterStrategyBelow(int threshold) : threshold(threshold) {}

    bool shouldPass(int value) override {
        return value < threshold;
    }
};
```



### 工厂模式

假设现在想要计算输入和的平均值，但是我们并不能对于所有输入提前获得输入的个数，这样一来我们不得不提供一个专门计算平均值的新目标函数

```cpp
int reduce(Inputer *inputer, Reducer *reducer) {
    int res = reducer.init();
    // 循环读取数据
    while (int tmp = inputer.fetch()) {
        res = reducer.add(res, tmp);
    }
    return res;
}

int average_reduce(Inputer *inputer, Reducer *reducer) {
    int res = reducer.init();
    int count = 0;
    
    // 循环读取数据
    while (int tmp = inputer.fetch()) {
        res = reducer.add(res, tmp);
        count++;
    }
    return res / count;
}
```

为了统一接口，我们实际上可以将计算状态也保存到 `Reducer` 中，只是最后获得计算结果。因此我们将 `Reducer` 的状态也分离出接口，用于保存循环计算的状态；而 `Reducer` 本身用于获得初始化状态

```cpp
struct ReducerState {
    virtual void add(int val) = 0;
    virtual int result() = 0;
};

struct Reducer {
    virtual unique_ptr<ReducerState> init() = 0;
};

int reduce(Inputer *inputer, Reducer *reducer) {
    unique_ptr<ReducerState> state = reducer->init();
    while (auto val = inputer->fetch()) {
        state->add(val);
    }
    return state->result();
}
```

这样平均计算器可以写成

```cpp
struct AverageReducerState : ReducerState {
    int res;
    int count;
	
    AverageReducerState() : res(0), count(0) {}

	// 在每步计算中更新状态
    void add(int val) override {
        res = res + val;
        count = count + 1;
    }

    int result() override {
        return res / count;
    }
};

struct AverageReducer : Reducer {
    unique_ptr<ReducerState> init() override {
        return make_unique<AverageReducerState>();
    }
};
```

普通的求和计算器不需要保存计数

```cpp
struct SumReducerState : ReducerState {
    int res;

    SumReducerState() : res(0) {}

    void add(int val) override {
        res = res + val;
    }

    int result() override {
        return res;
    }
};

struct SumReducer : Reducer {
    unique_ptr<ReducerState> init() override {
        return make_unique<SumReducerState>();
    }
};
```



使用 `init` 创建状态似乎没有必要，不过由于多态性质，这么做可以用于统一不同 `Reducer` 类型下，不同计算状态的创建过程。例如多线程计算

```cpp
int reduce(Inputer* inputer, Reducer* reducer) {
    std::vector<std::jthread> g;
    std::list<std::unique_ptr<ReducerState>> local_states;
    std::vector<int> chunk;
    
    // 用于创建线程队列
    auto enqueue_chunk = [&]() {
        // 添加空的状态节点
        local_states.emplace_back();

        // 捕获数据容器 chunk，捕获状态链表的新节点
        g.push_back(std::jthread([reducer, chunk = std::move(chunk), &back = local_states.back()]() {
            auto local_state = reducer->init();
            for(auto&& c: chunk) {
                local_state->add(c);
            }
            // 将新的状态添加到节点中（利用了 list 节点不会改变的特性）
            back = std::move(local_state);
        }));

        // 清空数据容器
        chunk.clear();
    };

    // 循环读取数据，凑足 64 个数据后创建一个线程进行处理
    while(auto tmp = inputer->fetch()) {
        if(chunk.size() < 64) {
            chunk.push_back(tmp.value());
        } else {
            enqueue_chunk();
        }
    }

    // 如果剩余数据不足 64 个，则也要创建一个线程进行处理
    if(chunk.size() > 0) {
        enqueue_chunk();
    }

    // 等待所有线程完成
    g.clear();

    // 合并所有线程计算的结果
    auto final_state = reducer->init();
    for(auto&& local_state: local_states) {
        final_state->add(local_state->result());
    }
    return final_state->result();
}
```



### 状态模式

状态模式用于一个对象需要在多种状态下进行切换，并且切换操作较为复杂的情况。如果使用枚举表示状态，就需要频繁修改状态转换函数，违背了开放封闭原则。例如 `Monster` 类需要在 `Idle` 和 `Attack` 状态之间切换，就可以将状态封装为接口，通过子状态实现接口调用。

```cpp
struct Monster;

struct State {
    virtual void update(Monster* m) = 0;
};

struct Monster {
    State* state = nullptr;

	// 刷新状态
    void update() { state->update(this); }

protected:
	// 状态转换函数
    void change_state(State* s) {
        delete state;
        state = s;
    }

    bool see_player() { return true; }
};

struct IdleState;
struct AttackState;

struct IdleState : State {
    void update(Monster* m) override {
        if(m->see_player()) {
            m->change_state(new AttackState());
        }
    }
};

struct AttackState : State {
    void update(Monster* m) override {
        if(!m->see_player()) {
            m->change_state(new IdleState());
        }
    }
};
```

当我们需要添加新的状态转换，只需要继承出新的状态子类即可，避免了修改原始代码。

> [!note]
> 注意到状态更新时删除了自身，这一操作是可行的。



### 原型模式

原型模式提供一个深拷贝成员函数 `clone`，它的作用是进行隐式的指针深拷贝

```cpp
struct Person {
    virtual Person* clone() const = 0;
};

struct Student : Person {
    Person* clone() const override {
        return new Student(*this);
    }   
};
```

这种写法的好处在于简化了多态指针的深拷贝过程

```cpp
Person* person = new Student();
Person* nPerson = new Student(*dynamic_cast<Student*>(person));	// 显式拷贝，需要写出转换的类型 Student
Person* mPerson = person->clone();								// 简化后的拷贝，隐式转换
```

现在更建议使用智能指针

```cpp
struct Person {
    virtual std::unique_ptr<Person> clone() const = 0;
};

struct Student : Person {
    std::unique_ptr<Person> clone() const override {
        return std::make_unique<Student>(new Student(*this));
    }   
};
```



### CRTP 模式

CRTP (Curiously Recurring Template Pattern) 是一种模板元编程技术，思路是将派生类作为模板参数传递给基类，从而实现一些自动化功能。例如通过基类函数提供子类的调用模板

```cpp
#include <iostream>

template <class Derived> struct Pet {
    void feed() {
        Derived* that = static_cast<Derived*>(this);
        that->speak();
    }
};

struct Dog : public Pet<Dog> {
    void speak() { std::cout << "Woof!" << std::endl; }
};

struct Cat : public Pet<Cat> {
    void speak() { std::cout << "Meow!" << std::endl; }
};

int main() {
    Dog dog;
    dog.feed();

    Cat cat;
    cat.feed();
    return 0;
}
```

> [!note]
> 使用 CRTP 模式可以避免使用虚函数，因此执行效率会更高。不过实际可以与虚函数相结合使用。



CRTP 模式与原型模式结合，实现自动化定义 clone 虚函数

```cpp
struct Ball {
    virtual std::unique_ptr<Ball> clone() = 0;
};

template <class Derived> struct BallImpl : public Ball {
    std::unique_ptr<Ball> clone() override { return std::unique_ptr<Ball>(new Derived(*static_cast<Derived*>(this))); }
};

// 不需要再写 clone() 函数，因为 BallImpl 已经实现了 clone() 函数
struct RedBall : public BallImpl<RedBall> {};
struct BlueBall : public BallImpl<BlueBall> {};
```



### 组件模式

组件模式用于为固定对象添加新的功能，并且这些功能会修改固定对象的状态。例如

```cpp
struct GameObject;

struct Component {
    virtual void update(GameObject* obj) = 0;
    virtual ~Component() = default;
};

struct GameObject {
    std::vector<Component*> components;

    void addComponent(Component* component) { components.push_back(component); }

    void update() {
        for(auto& component: components) {
            component->update(this);
        }
    }
};
```

这样可以将 `GameObject` 的属性拆分到不同的 `Component` 对象中，并且根据需要添加不同的属性和功能。



### 观察者模式

要实现不同组件之间相互访问、协同操作，就需要建立组件之间的通信。一种方案是使用 `typeid` 作为键保存不同的组件，这样保证组件的唯一性。通过 `getComponent` 可以从任意组件中访问其它组件，从而实现了组件通信。

```cpp
struct GameObject;

struct Component {
    virtual void update(GameObject* obj) = 0;
    virtual ~Component() = default;
};

struct GameObject {
    std::map<type_info, Component*> components;

	// 用于获得其它组件
    template <typename T> T* getComponent() {
        if(auto it = components.find(typeid(T)); it != components.end()) {
            return dynamic_cast<T*>(it->second);
        }
        return nullptr;
    }

    void addComponent(Component* component) { components[typeid(component)] = component; }

    void update() {
        for(auto& [type, component]: components) {
            component->update(this);
        }
    }
};
```



然而，我们可以将消息处理单独提取，由 `GameObject` 发送消息，在消息类中保存消息信息，在组件中接收消息并处理。

```cpp
struct GameObject;

struct Message {
    virtual ~Message() = default;
};

struct Component {
    virtual void update(GameObject* obj) = 0;
    virtual void handleMessage(Message* message) = 0;
    virtual ~Component() = default;
};

struct GameObject {
    std::map<type_info, Component*> components;

    template <typename T> T* getComponent() {
        if(auto it = components.find(typeid(T)); it != components.end()) {
            return static_cast<T*>(it->second);
        }
        return nullptr;
    }

    void addComponent(Component* component) { components[typeid(component)] = component; }

    void update() {
        for(auto& [type, component]: components) {
            component->update(this);
        }
    }

    void sendMessage(Message* message) {
        for(auto& [type, component]: components) {
            component->handleMessage(message);
        }
    }
};
```

但是这会将消息发送给所有组件，并且在组件内部还需要使用 `dynamic_cast` 动态转换并判断是否是可接收的消息，造成了性能浪费。



### 发布-订阅模式

发布-订阅模式是对观察者模式的升级，它通过一个事件总线管理消息分发。每个组件可以订阅需要的消息类型，事件总线只会将消息发布给订阅消息的组件。例如

```cpp
struct GameObject;

struct Message {
    virtual ~Message() = default;
};

struct Component {
    virtual void update(GameObject* obj) = 0;
    virtual void handleMessage(Message* message) = 0;
    virtual void subscribeMessage(GameObject* obj) = 0;
    virtual ~Component() = default;
};

struct GameObject {
    std::vector<Component*> components;
    std::map<type_info, std::vector<Component*>> subscribers;

	// 订阅组件
    template <class EventType> void subscribe(Component* component) { subscribers[typeid(EventType)].push_back(component); }

	// 发送消息
    template <class EventType> void sendEvent(EventType* event) {
        for(auto& component: subscribers[typeid(EventType)]) {
            component->handleMessage(event);
        }
    }

	// 添加组件，同时为组件订阅消息
    void addComponent(Component* component) {
        components.push_back(component);
        component->subscribeMessage(this);
    }

    void update() {
        for(auto& component: components) {
            component->update(this);
        }
    }
};

struct MoveMessage : Message {
    int x, y;
};

struct Movable : Component {
    void update(GameObject* obj) override { std::cout << "GameObject " << obj << " is moving" << std::endl; }

	// 处理消息
    void handleMessage(Message* message) override {
        if(MoveMessage* moveMessage = dynamic_cast<MoveMessage*>(message)) {
            // do something with moveMessage
        }
    }

	// 为组件订阅消息
    void subscribeMessage(GameObject* obj) override { obj->subscribe<MoveMessage>(this); }
};
```



### 访问者模式

使用发布-订阅模式仍然需要在处理消息时进行 `dynamic_cast`，为了提高效率，可以使用访问者模式。例如

1. 组件基类提供 `visit` 空实现，针对不同消息类型重载
2. 每个组件接收消息，然后调用消息的 `accept` 接口
3. 每个消息接收组件，然后调用组件的 `visit` 接口

通过来回调用接口，进入 `accept` 后，调用 `visit` 时参数 `this` 就已经是具体的消息类型，自动匹配到对应的组件重载函数，间接实现了类型转换。

```cpp
struct GameObject;
struct Component;

struct Message {
    virtual void accept(Component* visitor) = 0;
    virtual ~Message() = default;
};

struct MoveMessage : Message {
    int x, y;

    virtual void accept(Component* visitor) override { visitor->visit(this); }
};

struct JumpMessage : Message {
    int height;

    virtual void accept(Component* visitor) override { visitor->visit(this); }
};

struct Component {
	// 组件基类提供访问接口
    virtual void visit(MoveMessage* message) {}
    virtual void visit(JumpMessage* message) {}

    virtual void update(GameObject* obj) = 0;
    virtual void handleMessage(Message* message) = 0;
    virtual ~Component() = default;
};

struct GameObject {
    std::vector<Component*> components;

    void addComponent(Component* component) { components.push_back(component); }

    void update() {
        for(auto& component: components) {
            component->update(this);
        }
    }

    void handleMessage(Message* message) {
        for(auto& component: components) {
            component->handleMessage(message);
        }
    }
};

struct Movable : Component {
    void update(GameObject* obj) override { std::cout << "GameObject " << obj << " is moving" << std::endl; }

    void handleMessage(Message* message) override { message->accept(this); }

	// 具体的消息操作
    void visit(MoveMessage* message) {
    	// do something
    }
};
```

使用访问者模式时

- 每添加一个新的消息，就需要提供对应的 `visit` 重载。每个消息的 `accept` 函数都相同，造成了代码重复。
- 每添加一个新的组件，只需要实现这个组件的接口。

因此访问者模式常用于消息类型有限，需要频繁添加组件类型的场景。



利用 C++ 模板元编程技术，使用 CRTP 模式可以简化 `accept` 接口构建

```cpp
struct Message {
    virtual void accept(Component* visitor) = 0;
    virtual ~Message() = default;
};

template <class Derived> struct MessageImpl : Message {
    void accept(Component* visitor) override {
    	// 静态判断，要求继承关系
        static_assert(std::is_base_of<MessageImpl, Derived>::value, "Derived must be a Component");
        visitor->visit(static_cast<Derived*>(this));
    }
};

struct MoveMessage : MessageImpl<MoveMessage> {
    int x, y;
};

struct JumpMessage : MessageImpl<JumpMessage> {
    int height;
};
```



## 编译选项

开启编译时警告可以在编译期就检测出代码隐含的问题，避免许多细节上的错误。

| `-Winvalid-memory-model`          | 禁止无效的内存模型                        | `-Wsuggest-override`     | 建议重写虚函数         |
| `-Wattributes`                    | 禁止 `[[maybe-unused]]` 等属性写错      | `-Wmissing-declarations` | 禁止未声明的函数        |

其中 `error=` 表示将右侧的警告类型视为错误。我们以其中部分警告为例

- `-Wdelete-non-virtual-dtor` 会在虚函数继承时，对未删除的非虚析构函数报错，避免使用多态性质时，删除基类指针不调用子类析构函数的问题。对于智能指针，如果使用 `unique_ptr` 会发出警告，使用 `shared_ptr` 时不发出警告（因为 `shared_ptr` 存在默认删除器，它以子类指针作为参数，可以正确析构）
- `-Wparentheses` 用于检测类似 `if (x = 0)` 的错误
- `-Wreturn-type` 用于检测应当有返回值却无返回的函数，有返回值的函数不返回值是未定义行为
- `-Wreorder` 要求类成员的初始化顺序必须符合定义顺序

要得到更直接地规范代码，使用 `-Werror` 选项将所有警告视为错误。



## 未定义行为

1. 不能解引用空指针
2. 不能解引用 `end` 迭代器
3. `this` 不能为空（不要判断 `this == nullptr`）
4. `reinterpret_cast` 后以不兼容的类型访问
5. `union` 访问未激活的成员
6. 有符号整数的加减乘除溢出
7. 除数为 0
8. 返回类型不为 `void` 的函数
9. 函数指针调用时为空
10. 不能读取未初始化的变量
11. 指针加减法越过数组边界
12. 指针可以指向数组尾部（类似 `end` 迭代器），但不能解引用
13. 不能访问未初始化的指针
14. 不能访问已释放的内存
15. 不能访问已析构的对象
16. 不匹配 `new/delete, new[]/delete[], malloc/free`
17. `ctype.h` 中函数的字符参数只支持 ASCII 字符（范围 `0-127`）
18. `memcpy` 的 `src, dst` 不能为空指针
19. `memcpy` 的 `src, dst` 不能存在重叠，应当改为使用 `memmove`
20. 类型 `T` 指针必须对齐到 `alignof(T)`
21. 多线程访问同一个对象，且其中至少有一个写入访问
22. 同一线程中同一非 `recursive_mutex` 重复上锁
23. `vector` 的 `[]` 操作符越界
24. 容器迭代器失效
25. `v` 为空时访问 `v.back()`
