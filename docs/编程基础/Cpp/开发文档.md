# 开发文档

## C++ 规范

### 头文件

#### 防护机制

头文件应当具有防护机制，以防止多次包含。宏的命名按照路径，例如 `foo/src/bar/baz.h` 应当具有如下防护符号

```cpp
#ifndef FOO_BAR_BAZ_H_
#define FOO_BAR_BAZ_H_
...
#endif // FOO_BAR_BAZ_H_
```



#### 前向声明

尽可能避免前向声明，而是直接引入头文件。



前向声明尚未定义，例如在类、函数、变量定义前进行声明

```cpp
// In a C++ source file:
class B;
void FuncInB();
extern int variable_in_b;
ABSL_DECLARE_FLAG(flag_in_b);
```



优势：

* 前向声明可以节省编译时间，减少不必要的重复编译。
* 前向声明会隐藏依赖，允许用户在头文件修改时跳过重新编译。

缺陷：

* 前向声明会使自动工具难以发现定义符号的模块。
* 前向声明可能会被对库的后续更改破坏。函数和模板的前向声明可以防止标头所有者对其 API 进行其他兼容的更改，例如扩大参数类型、添加具有默认值的模板参数或迁移到新的命名空间。
* 命名空间中的前向声明导致未定义行为。
* 可能很难确定是否需要前瞻声明或完整声明。
* 在一个头文件中进行多个前向声明会更繁琐。
* 构建代码以启用前向声明（例如，使用指针成员而不是对象成员）可能会使代码更慢、更复杂。



#### 内联函数

仅当函数较小（例如，10 行或更少）时，才以内联方式定义函数。



优势：

* 内联函数可以生成更有效的目标代码，只要内联函数很小。可以随意使用内联访问器和突变体，以及其他简短的性能关键型功能。

缺陷：

* 过度使用内联实际上会使程序变慢。根据函数的大小，内联它可能会导致代码大小增加或减少。内联非常小的访问器函数通常会减小代码大小，而内联非常大的函数会显著增加代码大小。在现代处理器上，由于更好地使用指令缓存，较小的代码通常运行得更快。

原则：

* 如果函数的长度超过 10 行，则不要内联该函数。当心析构函数，由于隐式成员和基析构函数调用，它们通常比它们看起来更长！
* 使用循环或 switch 语句的函数不进行内联。

即使函数被声明为内联，它们也并不总是内联的；例如，虚拟函数和递归函数通常不内联。



#### 包含名称及顺序

头文件包含顺序：相关头文件、C 系统头文件、C++ 标准库头文件、其它库头文件、项目头文件。



例如 `/src/foo/internal/fooserver.cc` 中的头文件包含顺序

```cpp
// 与 fooserver.cc 直接相关的头文件
#include "foo/server/fooserver.h"

// C 头文件
#include <sys/types.h>
#include <unistd.h>

// C++ 头文件
#include <string>
#include <vector>

// 项目头文件
#include "base/basictypes.h"
#include "base/commandlineflags.h"
#include "foo/server/bar.h"
```

有时，与系统相关的代码需要条件包含。这样的代码可以将条件包含放在其他包含之后。例如

```cpp
#include "foo/public/fooserver.h"

#include "base/port.h"  // For LANG_CXX11.

#ifdef LANG_CXX11
#include <initializer_list>
#endif  // LANG_CXX11
```



### 作用域

#### 命名空间

除少数例外情况外，请将代码放在命名空间中。命名空间应具有基于项目名称的唯一名称，并可能基于其路径。不要使用 using-指令。不要使用内联命名空间。



优势：

* 命名空间提供了一种防止大型程序中名称冲突的方法，同时允许大多数代码使用合理的短名称。
* 内联命名空间自动将其名称放在封闭作用域中。例如

```cpp
namespace outer {
inline namespace inner {
  void foo();
}  // namespace inner
}  // namespace outer
```

内联命名空间主要用于实现跨版本的兼容性。

缺陷：

* 命名空间可能会令人困惑，因为它们使弄清楚名称引用的定义机制变得复杂。
* 内联命名空间可能造成混淆，因为名称实际上并不局限于声明它们的命名空间。它们仅作为某些较大版本控制策略的一部分有用。
* 在某些情况下，有必要通过符号的完全限定名称反复引用符号。对于深度嵌套的命名空间，这可能会增加很多混乱。



使用方法：

* 遵循命名空间名称的规则。
* 使用注释终止多行命名空间。
* 在包含、gflags 定义/声明和来自其它命名空间的类的前向声明之后，命名空间应当包含整个源文件

```cpp
// In the .h file
namespace mynamespace {

// All declarations are within the namespace scope.
// Notice the lack of indentation.
class MyClass {
public:
    ...
    void Foo();
};
} // namespace mynamespace
```

在源文件中也应当包含所有定义

```cpp
// In the .cc file
namespace mynamespace {
// Definition of functions is within scope of the namespace.
void MyClass::Foo() {
...
}
} // namespace mynamespace
```

* 不要在命名空间中声明任何内容，包括标准库类的前向声明。在命名空间中声明实体是未定义的行为，即不可移植。若要从标准库声明，请包括相应的头文件。
* 不能使用 using 使命名空间中的所有名称都可用。

```cpp
// Forbidden -- This pollutes the namespace.
using namespace foo;
```

* 不要在头文件中的命名空间范围内使用命名空间别名，除非在显式标记的仅限内部的命名空间中使用，因为导入到头文件中的命名空间中的任何内容都将成为该文件导出的公共 API 的一部分。

```cpp
// Shorten access to some commonly used names (in a .h file).
namespace librarian {
namespace impl {
// 内部可以使用别名
namespace sidetable = ::pipeline_diagnostics::sidetable;
} // namespace impl
inline void my_inline_function() {
// 函数内部也可以使用
namespace baz = ::foo::bar::baz;
...
}
} // namespace librarian
```

* 不要使用内联命名空间。



#### 内部链接

对于 `.cc` 文件中不需要从外部引用的定义，使用未命名空间将其包含。这样如果两个不同的文件声明了相同的内容，它们依然是独立的。

```cpp
namespace {
	...
} // namespace
```

不要在 `.h` 文件中使用内部链接。



#### 非成员、静态成员和全局函数

建议将非成员函数放在命名空间中，少使用完全全局函数。不要简单地使用类对静态成员进行分组。类的静态方法通常应与类的实例或类的静态数据密切相关。



优势：

* 将非成员函数放在命名空间中可避免污染全局命名空间。

缺陷：

* 非成员和静态成员函数作为新类的成员可能更有意义，特别是当它们访问外部资源或具有重要依赖关系时。

使用方法：

* 非成员函数不应依赖外部变量，并且应几乎总是存在于命名空间中。
* 不要创建类只为了给静态成员进行分组，这与使用通用前缀没有什么不同。
* 如果定义了非成员函数，并且仅在其文件中需要该函数，使用内部链接限制其作用域。



#### 局部变量

应当将函数变量放在尽可能小的作用域中，并在声明时初始化变量。

```cpp
int i;
i = f(); // Bad -- initialization separate from declaration.
int j = g(); // Good -- declaration has initialization.
```

如果变量是一个对象，在循环过程中会反复调用构造函数和析构函数

```cpp
// Inefficient implementation:
for (int i = 0; i < 1000000; ++i) {
    Foo f; // My ctor and dtor get called 1000000 times each.
    f.DoSomething(i);
}
```

将循环中使用的变量放在外部会更高效

```cpp
Foo f; // My ctor and dtor get called once each.
for (int i = 0; i < 1000000; ++i) {
    f.DoSomething(i);
}
```



#### 静态变量和全局变量

禁止使用具有静态储存持续时间（static 修饰）的对象，除非它们具有平凡析构，即析构函数不执行任何操作。更正式地说，该类型没有用户定义的或虚析构函数，所有基类和非静态成员都可析构。静态本地变量可以使用动态初始化（new）。不建议对静态类成员变量或命名空间范围的变量使用动态初始化，但在有限的情况下允许使用。



优势：

* 全局和静态变量对于大量应用程序非常有用。

缺陷：

* 使用动态初始化或具有非平凡析构函数的全局和静态变量会产生复杂性，导致难以找到的错误。



##### 销毁

只对于具有平凡析构的对象使用静态类型。基本类型都是平凡析构，它们的数组也是平凡析构，例如

```cpp
const int kNum = 10; // allowed
struct X { int n; };
const X kX[] = {{1}, {2}, {3}}; // allowed

void foo() {
    static const char* const kMessages[] = {"hello", "world"}; // allowed
}

// allowed: constexpr guarantees trivial destructor
constexpr std::array<int, 3> kArray = {{1, 2, 3}};
```

注意标有 `constexpr` 的变量具有平凡析构。相对的，大部分对象都不是平凡析构

```cpp
// bad: non-trivial destructor
const std::string kFoo = "foo";

// bad for the same reason, even though kBar is a reference (the
// rule also applies to lifetime-extended temporary objects)
const std::string& kBar = StrCat("a", "b", "c");
void bar() {
    // bad: non-trivial destructor
    static std::map<int, int> kData = {{1, 0}, {2, 0}, {3, 0}};
}
```

引用不是对象，因此它们不受此约束，但是对动态初始化的约束仍然适用。特别地，可以对动态内存进行本地静态引用

```cpp
static T& t = *new T;
```



##### 初始化

常量初始化应当声明常量构造函数。例如下面是有问题的初始化

```cpp
// Some declarations used below.
time_t time(time_t*); // not constexpr!
int f(); // not constexpr!
struct Bar { Bar() {} };

// Problematic initializations.
time_t m = time(nullptr); // initializing expression not a constant expression
Foo y(f()); // ditto
Bar b; // chosen constructor Bar::Bar() not constexpr
```

正确的写法是

```cpp
struct Bar { constexpr Bar() {} };

// 通过常量构造函数创建常量初始化
constexpr Bar b;
```



##### 常见模式

* 全局字符串：如果需要全局或静态字符串常量，考虑使用 `constexpr` 字符数组或字符指针，指向字符串文本。
* 映射、集合和其他动态容器：如果需要静态的固定集合，不能将标准库中的动态容器用作静态变量，因为它们具有重要的析构函数。考虑使用平凡类型的数组。如果确实需要动态容器，应当使用本地静态指针。
* 智能指针：智能指针在销毁过程中执行清除，因此不能作为全局或静态变量。最好使用普通指针，并且永远不要删除它。
* 自定义类型的静态变量：如果需要自定义，则需要指定普通析构函数和 `constexpr` 构造函数。
* 如果这些方法都失败了，可以动态创建对象，并且不使用函数局部静态指针或引用。例如

```cpp
static const auto& impl = *new T(args...);
```



#### thread_local 变量

不在函数中定义的局部线程变量必须使用编译时常量进行初始化，优先于其它定义局部线程变量的方法。



可以使用说明符声明变量

```cpp
thread_local Foo foo = ...;
```

这样的变量实际上是对象的集合，当不同的线程访问它时，实际上在访问不同的对象。它们与静态变量类似，可以在命名空间范围、函数内部或作为静态类成员声明，但不能作为普通类成员声明。



优势：

* 局部线程变量的初始化与静态变量类似，但是必须针对每个线程单独初始化。这意味着在函数中声明的变量是安全的，但其它变量会受到与静态变量相同的初始化顺序问题影响。
* 在其线程终止之前不会被销毁，因此不存在静态变量的销毁顺序问题。
* 它是创建局部线程变量的唯一标准支持。

缺陷：

* 访问变量可能会触发执行不可预测且无法控制的其它代码。
* 它实际上是全局变量，具有全局变量的所有缺点。
* 变量消耗的内存随着线程数增加。
* 非静态成员不能是局部线程变量。
* 如果变量具有复杂的析构函数，可能导致释放后使用的错误。



局部线程变量如果位于类或命名空间中，必须使用编译时的常量进行初始化。使用 `ABSL_CONST_INIT` 来强制执行

```cpp
ABSL_CONST_INIT thread_local Foo foo = ...;
```

函数内的局部线程变量没有安全问题，但是在线程退出期间仍存在释放后使用的风险。可以使用函数作用域模拟类或命名空间的作用域，例如

```cpp
Foo& MyThreadLocalFoo() {
    thread_local Foo result = ComplicatedInitialization();
    return result;
}
```

通过静态引用返回创建具有局部作用域的全局变量。每当线程退出时，局部线程变量都会被销毁。如果此类变量的析构函数引用了任何其他变量，可能造成难以发现的释放后使用错误。



### 类

#### 构造函数

应当避免在构造函数中调用虚方法，避免在初始化失败时无法报错。



优势：

* 无需担心类是否已经初始化。
* 通过调用构造函数初始化可能易于与标准容器或算法一起使用。

缺陷：

* 如果调用虚函数，则这些调用将不会在子类中实现。将来对类的修改可能会引入此问题。
* 没有简单的方法报错，除了使程序崩溃或使用异常（禁止）。
* 如果初始化失败，需要通过状态检查机制（例如 `bool IsValid()`）判断。
* 不能获得构造函数的地址，因此构造函数中完成的任何工作不能轻易地传递给另一个线程。



#### 隐式转换

不要定义隐式转换，将 explicit 关键字用于转换运算符和单参数构造。



隐式转换允许在需要不同类型（目标类型）的情况下使用一种类型（源类型）的对象，例如将 int 参数传递给 double 参数的函数。使用 explicit 关键字应用于构造函数或转换运算符，以确保仅当目标类型显式使用时才能使用它。例如

```cpp
class Foo {
    explicit Foo(int x, double y);
    ...
};
void Func(Foo f);

Func({42, 3.14}); // Error
```

这里表明 Foo 对象的构造必须显式进行。



优势：

* 隐式转换可以消除显式命名类型的需要。
* 隐式转换可以是重载的简单替换方法。
* 列表初始化语法是一种简洁有力的初始化方式。

缺陷：

* 隐式转换会隐藏类型不匹配的错误。
* 隐式转换会使代码更难阅读，尤其是重载的情况下。
* 采用单个参数的构造函数可能意外地使用隐式类型转换。因此无法判断它是否用来隐式转换。
* 隐式转换可能导致调用点不明确，尤其是存在双向隐式转换时。
* 如果目标类型是隐式的，则列表初始化可能出现问题，尤其是列表只有一个元素时。

使用方法：

* 必须在类定义中标记类型转换运算符和单个参数的构造函数。复制和移动构造函数不执行类型转换，因此不需要修饰。
* 隐式转换有时对于设计为可互换的类型时是必要的。
* 不能使用单个参数调用的构造函数和采用单个 `std::initializer_list` 参数的构造函数可以省略 explicit 修饰。



#### 可复制和可移动类型

类的公共 API 必须明确该类是可复制的、仅移动的，还是不可复制的或不可移动的。



优势：

* 可复制和可移动类型的对象可以按值传递和返回，这使得 API 更简单、更安全、更通用。与通过指针或引用传递对象不同，不存在所有权、生存期、可变性和类似问题混淆的风险。
* 可以防止客户端与实现之间的非本地交互，从而更容易被编译器理解、维护和优化。
* 此类对象可与需要按值传递的通用 API 一起使用，例如大多数容器，并且在类型组合方面提供额外的灵活性。
* 复制/移动构造函数和赋值运算通常比自定义方法更容易正确定义，因为它们可由编译器隐式生成，可以使用 `=default` 定义。它们通常也更有效，因为不需要堆分配内存或单独的初始化和分配步骤，并且它们有资格进行优化。
* 移动操作允许隐式且高效地将资源从右值对象中转移出去。

缺陷：

* 某些类型不需要可复制。对具有多态的基类执行复制操作非常危险，默认的复制操作可能不正确。
* 复制构造函数隐式调用，可能导致调用过程被遗漏。

使用方法：

* 每个类的公共接口都必须明确该类支持哪些复制和移动操作。

```cpp
class Copyable {
 public:
  Copyable(const Copyable& other) = default;
  Copyable& operator=(const Copyable& other) = default;

  // 上面对复制构造的声明抑制了隐式的移动构造，如果需要移动构造，就要显式地声明
};

class MoveOnly {
 public:
  MoveOnly(MoveOnly&& other) = default;
  MoveOnly& operator=(MoveOnly&& other) = default;

  // 上面对移动构造的声明隐式地删除了复制构造，当然我们也可以显式地写出来：
  MoveOnly(const MoveOnly&) = delete;
  MoveOnly& operator=(const MoveOnly&) = delete;
};

class NotCopyableOrMovable {
 public:
  // 不可复制、不可移动
  NotCopyableOrMovable(const NotCopyableOrMovable&) = delete;
  NotCopyableOrMovable& operator=(const NotCopyableOrMovable&)
      = delete;

  // 上面的声明隐式地禁止了移动操作，当然我们也可以显式地写出来：
  NotCopyableOrMovable(NotCopyableOrMovable&&) = delete;
  NotCopyableOrMovable& operator=(NotCopyableOrMovable&&)
      = delete;
};
```

只有当这些声明或删除显而易见时，才可以省略它们：

* 如果类没有 `private` 部分，例如 `struct` 或者只有接口的基类，则可复制性/可移动性可以通过 `public` 数据成员的性质来确定。
* 如果基类显然不可复制或不可移动，则派生类自然同理。
* 如果显式声明或删除复制构造函数或赋值操作，则另一种复制操作就不明显，应当具体声明；移动操作同理。



可复制类型的移动操作是一种性能优化，并且是 bug 和复杂性的潜在来源，因此要避免定义它们，除非能极大地提高效率。最好为基类设置 `protected` 构造函数和析构函数，或者提供纯虚函数使基类抽象化。避免从具体类中派生。



#### 结构与类

仅携带数据的被动对象使用 `struct`，其它都使用 `class` 。允许 `struct` 中提供构造函数、析构函数和帮助方法。



#### 结构与 Pair 或 Tuple

每当元素可以具有有意义的名称时，应当使用 `struct` 而不是 `pair` 或 `tuple` 。



#### 继承

组合通常比继承更合适。使用继承时，声明为 `public` 。



优势：

* 继承通过重用基类代码来减小代码大小。由于继承是编译时声明，因此可以由编译器检查错误。接口继承可以强制类公开特定 API，编译器可以检查派生类未定义 API 的必要方法时的错误。

缺陷：

* 由于实现子类的代码分布在基类和子类之间，因此可能更难理解实现。子类不能重写非虚拟函数，因此子类不能更改实现。
* 多继承存在许多问题，通常会带来更高的性能开销，并且可能导致“菱形”继承模式的风险。

使用方法：

* 所有继承都应该是 `public`，除非将基类的实例作为派生类的成员，才使用 `private` 继承。如果不打算将类作为基类，可以在类声明前添加 `final` 修饰。
* 不要过度使用继承，使用组合更合适。只有当子类能够被称为基类的一个特例时，才使用继承。
* 数据成员应当为 `private`；需要子类访问的成员函数应当避免使用 `protected` 。
* 使用 `final` 或 `override` 显式注释对虚函数或许析构函数的重载，注意重载时不要声明 `virtual` 。因为标记为 `override` 或 `final` 并且不是对基类虚函数重载的函数将不会编译，这有助于捕获常见错误。
* 允许多重继承，但是尽量不要进行多重实现的继承。



#### 操作符重载

明智地使用操作符重载。



优势：

* 操作符重载可以使用户定义类型的行为与内置类型相同，从而简化代码。

缺陷：

* 提供正确、一致且清晰的操作符重载需要谨慎，否则可能导致混淆和错误。
* 过度使用操作符可能导致代码混淆，尤其是重载语义不符合约定时。
* 操作符可能在直觉上被认为是简单操作，导致随意使用时隐含的高额开销。
* 如果重载的参数类型错误，可能会隐式调用另一个重载函数，导致错误难以发现。
* 重载 `&` 等一元操作符或 `&&`,``||``,`,` 可能出现问题。
* 操作符通常在类外定义，因此存在不同文件引入同一运算的不同定义的风险。

使用方法：

* 只有当重载运算符的含义显而易见且与内置运算一致时，才进行重载。
* 只在自定义的类型上定义操作符，避免定义为模板。
* 建议将 `const` 修饰的二元运算符定义为非成员函数。如果它定义为成员函数，则隐式转换只应用于右侧参数。例如 `a+b` 调用 `a` 的 `+` 运算，将 `b` 作为参数传入，隐式转换只作用于 `b` 。
* 不要刻意避免运算符重载。例如使用 `==` 优先于 `Equals()`；反之，不要仅仅因为其它库需要运算符重载而进行重载。
* 不要重载 `&&`,`||`,`,`,`&` 运算符。



#### 存取控制

类成员数据应当设为 `private`，除非它们是常量。代价是必要时提供 `const` 修饰的访问器。



#### 声明顺序

类定义通常以 `public:` 开头，然后是 `protected:`，然后是 `privated:`，如果某个部分是空的，就省略。每个部分中相似类型的声明组合在一起，按照如下顺序：

* 类型和类型别名；
* （可选，仅适用于 `struct`）非 `static` 数据成员；
* 静态常量；
* 工厂函数；
* 构造函数和赋值运算；
* 析构函数；
* 其它所有函数；
* 其它所有成员。

不要将大型方法定义内联到类定义中。通常，只有微不足道或非常短且对性能很重要的方法需要内联。



### 函数

#### 输入输出

* 首选通过返回值输出而不是参数输出：它们提高了可读性，并通常提供相同或更好的性能。
* 最好按值返回，否则按引用返回。避免返回指针，除非它可以为 `null` 。
* 必选输入参数通常应为值或 `const` 引用，必选输出和输入/输出参数通常为引用（不能为空）。通常使用 `std::optional` 表示可选的按值输入，如果必选参数为引用时使用 `const` 指针。使用非 `const` 指针表示可选输出和可选输入/输出参数。
* 避免定义需要超出生命周期的 `const` 引用参数的函数。
* 对函数参数排序时，所有仅输入的参数放在所有输出参数的前面。



#### 编写短函数

建议小而集中的函数，如果一个函数超过 40 行，应当考虑将其分解。



#### 函数重载

可以写一个函数接受 `const std::string &` 然后重载使其接受 `const char *`，但更建议使用 `std::string_view` 替代。

```cpp
class MyClass {
 public:
  void Analyze(const std::string &text);
  void Analyze(const char *text, size_t textlen);
};
```



优势：

* 重载可以允许同名函数采用不同参数来使代码更直观。

缺陷：

* 如果一个函数仅被参数类型重载，可能需要了解 C++ 的复杂匹配规则才能知道发生了什么。
* 如果派生类仅重载函数的某些变体，则会导致代码难以理解。

使用方法：

* 当变体之间没有语义差异时，可以使用重载。



#### 默认参数

当保证默认值始终具有相同的值时，允许在非虚函数上使用默认参数。



优势：

* 使用默认参数的函数允许偶尔覆盖默认值。与重载函数相比，默认参数具有更简洁的语法，并提供了“必选”和“可选”之间的明确区别。

缺陷：

* 默认参数是实现重载函数语义的另一种方法，因此不重载函数的所有原因都适用。
* 虚函数中参数的默认值由目标对象的静态类型确定，不能保证该函数的所有重载都声明相同的默认值。
* 默认参数在每个调用节点重新计算，可能使生成的代码膨胀。
* 函数指针存在默认参数时会造成混淆，因为函数签名通常与调用签名不匹配。使用函数重载可以避免这些问题。

使用方法：

* 禁止在虚函数中使用默认参数。
* 除非默认参数可以提高函数声明的可读性，否则使用函数重载。



#### 尾随返回类型

C++ 允许两种不同形式的函数声明

```cpp
int foo(int x);
auto foo(int x) -> int;
```

新的语法允许使用 `auto` 返回，然后尾随返回类型。



优势：

* 尾随返回类型时显式指定 lambda 表达式返回类型的唯一方式。
* 有时在参数列表出现后指定返回类型会更易读。尤其是在模板参数中

```cpp
template <typename T, typename U>
decltype(declval<T&>() + declval<U&>()) add(T t, U u);
```

这时通过尾随返回会更加清晰

```cpp
template <typename T, typename U>
auto add(T t, U u) -> decltype(t + u);
```

使用方法：

* 大多数情况下应当使用旧版的函数声明，仅在需要时使用尾随返回类型。



### C++ 特性

#### 右值引用

右值引用是一种只能绑定到临时对象的引用类型。例如声明

```cpp
void f(std::string&& s);
```

当标记 `&&` 应用于函数参数中的非限定模板参数时，将应用特殊的模板参数推导规则。此引用称为转发引用。



优势：

* 定义移动构造函数，可以移动值而不是复制值，可能导致性能的重大改进。

```cpp
// 移动构造只会导致简单的指针操作，而非复制大量数据
std::vector<std::string> v1;
auto v2(std::move(v1));
```

* 右值引用使实现可移动但不可复制的类型成为可能。例如有些类不应当被复制，但是需要作为参数传递或放在容器中时。
* 对于 `std::unique_ptr` 等标准库类型是必要的。
* 可以将参数转发到另一个函数，并且无论其参数是否为临时对象或常量都有效。这称为完美转发。

缺陷：

* 右值引用未被广泛理解，引用折叠和转发引用的特殊规则有些模糊。
* 右值引用常被误用。

使用方法：

不要使用右值引用，除非有以下情况：

* 定义移动构造和移动赋值；
* 将转发引用于 `std::forward` 结合使用，以支持完美转发；
* 定义成对的重载，例如 `Foo&&` 和 `const Foo&` 两种参数。



#### 友元

友元通常应在同一文件中定义，以便读者不必在另一个文件中查找类的私有成员的用法。常见的用法是对 `Foo` 声明 `FooBuilder` 为友元，这样后者可以正确构造 `Foo` 的内部状态。



#### 异常

不使用 C++ 异常。



优势：

* 异常允许更高级别的应用程序决定如何处理深度嵌套函数中不应该发生的故障。
* 大多数现代语言都使用异常。
* 某些第三方 C++ 库使用异常，关闭它们会导致库的集成变得困难。
* 异常是构造函数失败的唯一途径。我们可以使用工厂函数或 `Init()` 方法来模拟，但是这些分别需要堆分配或新的“无效”状态。
* 异常在测试框架中非常方便。

缺陷：

* 使用 `throw` 添加到现有函数时，必须检查其所有传递调用方。它们要么必须至少做出基本的异常安全保证，要么永远不会发现异常导致程序中断。例如 `f` 调用 `h`，`h` 调用 `g`，而 `h` 抛出异常被 `f` 捕获，此时 `g` 中代码就需要谨慎处理，防止内存没有正确释放。
* 异常使程序的控制流难以通过查看代码来评估：函数可能在意想不到的地方返回。
* 异常安全需要大量的编程实践。
* 开启异常会向生成的每个二进制文件添加数据，从而增加编译时间，并可能增加地址空间压力。
* 异常的可用性可能会鼓励开发人员在不合适的时候抛出，或者在不安全时从中恢复。

使用方法：

* 从表面上看，使用异常的好处大于成本，尤其是在新项目中。但是，对于现有代码，引入异常会对所有依赖代码产生影响。如果异常可以传播到新项目之外，那么新项目集成到现有的无异常代码中也会成为问题。



#### noexcept

`noexcept` 说明符用于指定函数是否会引发异常。如果异常从标记为 `noexcept` 的函数中产生，程序将通过 `std::terminate` 崩溃。修饰表达式的 `noexcept` 运算符执行编译时检查，如果表达式没有异常，则会返回 `true` 。



优势：

* 在某些情况下，将移动构造指定为 `noexcept` 可以提高性能，例如若 `T` 的移动构造函数声明为 `noexcept`，则 `std::vector<T>::resize()` 将会移动构造而非复制。
* 在启用异常的环境中，指定 `noexcept` 可以触发编译器优化。

缺陷：

* 如果禁用了异常，则使用 `noexcept` 不能保证正确。
* 很难撤销 `noexcept`，因为它以难以检测的方式消除了调用者可能依赖的保证。

使用方法：

* 如果 `noexcept` 准确地反映了函数的预期语义，则可以在对性能有用时使用。
* 如果完全禁用异常，则首选无条件 `noexcept` 。



#### 运行时类信息

运行时类信息（RTTI）允许在运行时查询对象的 C++ 类，这是通过使用 `typeid` 或 `dynamic_cast` 来完成的。



优势：

* RTTI 的标准替代方案需要修改或重新设计所讨论的类层次结构。有时，这种修改是不可行的。
* RTTI 在某些单元测试中很有用。例如在工厂类的测试中。它在管理对象与其模拟之间的关系时也很有用。
* RTTI 在考虑多个抽象对象时很有用。例如

```cpp
bool Base::Equal(Base* other) = 0;
bool Derived::Equal(Base* other) {
  Derived* that = dynamic_cast<Derived*>(other);
  if (that == nullptr)
    return false;
  ...
}
```

缺陷：

* 在运行时查询对象的类型通常意味着类层次结构的设计问题。
* RTTI 的无节制使用使代码难以维护。可能导致基于类型的决策树或 switch 语句分散在整个代码中，在进一步修改时必须检查所有语句。

使用方法：

* 可以在单元测试中自由使用，但在其它代码中尽可能避免使用。如果需要编写基于类的不同行为的代码，应该考虑替代方案：
    * 首选虚方法；
    * 如果要在对象外部处理，考虑双重调度方法，例如 Visitor 设计模式。
* 当程序的逻辑保证基类的给定实例实际上是特定派生类的实例时，可以在对象上自由使用 `dynamic_cast` 。这种情况通常可以考虑 `static_cast` 作为替代。
* 基于类型的决策树强烈说明代码问题

```cpp
if (typeid(*data) == typeid(D1)) {
  ...
} else if (typeid(*data) == typeid(D2)) {
  ...
} else if (typeid(*data) == typeid(D3)) {
...
```

* 当将其它子类添加到类层次结构中时，此类代码通常会中断。此外，当子类的属性发生更改时，很难找到并修改所有受影响的代码段。
* 不要手动实现类似 RTTI 的解决方法。



#### 类型转换

C++ 引入了不同于 C 的强制转换系统，用于区分强制转换操作的类型。



优势：

* C 转换的问题在于操作的模糊性：有时在进行转换 `(int)3.5`，有时进行强制转换 `(int)"hello"` 。括号初始化和 C++ 强制转换通常有助于避免这种歧义。此外，代码中的 C++ 强制转换更容易被搜索到。

缺陷：

* C++ 强制转换语法冗长且繁琐。

使用方法：

通常不要使用 C 样式强制转换。当需要显式类型转换时，使用 C++ 样式的强制转换。

* 使用大括号初始化来转换算术类型 `int64_t{x}` 。这是最安全的方法，因为如果转换可能导致信息丢失，那么编译就不能通过。
* 用 `absl::implicit_case` 安全地向上转换类型层次结构。
* 当需要显式强制转换时，使用 `static_cast` 执行与 C 样式等价的强制转换。
* 使用 `const_cast` 来移除 `const` 限定符。
* 用 `reinterpret_cast` 对指针类型以及整数到其它类型的不安全转换。



#### Streams

在适当的情况下使用 steam，并坚持简单的用法。仅对表示值的类型重载 `<<` 操作符，且只写入用户可见的值，而不写入任何实现细节。



优势：

* 流式操作符提供易学习、可移植、可重用和可扩展的 API，相较之下 `printf` 甚至不支持 `std::string`，很难可移植使用。
* 通过 `std::cin,std::cout,std::cerr,std::clog` 提供控制台 I/O 的一流支持。

缺陷：

* 可以通过改变流的状态，这种改变是持续性的，因此代码行为可能收到先前操作的影响。
* 由于上述问题，代码和数据在流中的混合方式以及运算符重载的使用很难精确控制输出。
* 通过 `<<` 建立的输出的方式将输出顺序融入到代码中，并且对本地化支持有限。
* 许多 `<<` 重载会给编译器带来很高的解析和语义分析的成本。

使用方法：

* 仅当流是工作的最佳工具时，才使用流。通常日志记录库是比 `std::cerr,std::clog` 更好的选择。
* 避免将流用于面向外部用户或处理不受信任数据的 I/O 。
* 避免使用流 API 中有状态部分（错误状态除外），例如 `imbue(),xalloc(),register_callback()` 。使用显式格式设置函数，例如 `absl::StreamFormat()` 而不是流操作器或格式设置标志。
* 仅当类型表示值时，才重载 `<<` 运算符，并且只输出该值，不应暴露实现细节。如果需要打印对象内部进行调试，定义 `DebugString()` 函数。



#### 递增和递减

使用递增和递减运算符的前缀形式 `++i`，除非需要后缀语义。



优势：

* 后缀递增/递减表达式的计算结果与修改前相同，可能导致代码更紧凑但是更难阅读。
* 前缀形式通常更具可读性，效率永远不会降低，并且可以更搞钱，因为不需要复制值。

缺陷：

* 在 C 语言传统中使用后缀增量，尤其是在 `for` 循环中。



#### const

在 API 中，只要有意义就使用 `const`；对于 `const` 的某些用途，使用 `constexpr` 是更好的选择。



优势：

* 更容易理解变量的使用方式。
* 允许编译器进行更好的类型检查，并生成更好的代码。
* 帮助人们了解多线程程序中无需锁定即可安全使用的函数。

缺陷：

* 如果将一个 `const` 变量传递给一个函数，该函数原型中必须有一个 `const`，这对于调用库函数可能产生问题。

使用方法：

* 强烈建议在有意义和准确的地方使用 `const` 。
* 对于按值传递的函数参数，不建议使用 `const` 声明。
* 除非方法改变了对象的逻辑状态，否则使用 `const` 声明。



#### constexpr, constinit, consteval

优势：

* 使用 `constexpr` 可以定义具有浮点表达式的常量，而不仅仅是文字；定义用户定义类型的常量；以及使用函数调用定义常量。

缺陷：

* 过早将某些内容标记为 `constexpr` 可能会导致后期修改时的迁移问题。

使用方法：

* 用 `constexpr` 定义真常量（编译/链接时固定）或确保常量初始化。
* 用 `constinit` 确保非常量变量的常量初始化。
* 用 `consteval` 定义不得在运行时调用的代码。
* 不要使用 `constexpr` 或 `consteval` 强制内联。



#### 预处理宏

避免定义宏，尤其是在头文件中。首选内联函数、枚举和 `const` 变量。使用与项目相关的前缀命名宏。不要使用宏定义 C++ API 片段。



缺陷：

* 宏意味着看到的代码与编译器看到的代码不同，可能会引入意外行为，特别是宏具有全局范围。
* 当宏用于 C++ API 片段时，问题更为严重。当开发人员错误地使用接口时，编译器发出的每条错误都必须解释宏是如何形成接口的。例如避免以下模式

```cpp
class WOMBAT_TYPE(Foo) {
  // ...

 public:
  EXPAND_PUBLIC_WOMBAT_API(Foo)

  EXPAND_WOMBAT_COMPARISONS(Foo, ==, <)
};
```

使用方法：

* 不要使用宏内联性能关键代码，而应使用内联函数。
* 不要使用宏来储存常量，而应使用 `const` 变量。
* 不要使用宏“缩写”长变量名，而是使用引用。
* 不要在 .h 文件中定义宏。
* 在使用之前 `#define` 宏，使用之后 `#undef` 宏。
* 在使用自定义宏替换现有宏时，不要只 `#undef` 现有宏，而是选择一个可能唯一的名称。
* 尽量不要使用扩展到不平衡的 C++ 构造的宏。
* 不使用 `##` 生成函数/类/变量名。

强烈建议不要从头文件导出宏，即在头文件末尾没有执行 `#undef` 操作。如果需要导出宏，确保宏具有全局唯一名称。



#### 0 和 `nullptr/NULL`

使用方法：

* 对于指针使用 `nullptr`，这提供了类型安全。
* 对于字符使用 `'\0'` 而不是 `0` 结尾。使用正确的类型可使代码更具可读性。



#### sizeof

在获取特定变量的大小时使用 `sizeof(var)` 而不是 `sizeof(type)` 。如果有人更改变量类型，`sizeof(var)` 将会自动更新。只对于与任何特定变量无关的代码使用 `sizeof(type)` 。



#### 类型推导

仅当类型推导能够让不熟悉代码的读者更熟悉代码，或者让代码更安全时，才使用类型推导。不要仅仅为了避免编写显式类型而使用。



优势：

* C++ 类型名称可能很长且繁琐，尤其是设计模板或命名空间时。
* 当 C++ 类型名称在单个声明或小代码区域中重复时，类型推导有助于提高可读性。
* 有时让类型被推导会更安全，因为可以避免意外复制或类型转换的可能性。

缺陷：

* 当类型是显式的时，C++ 代码通常更清晰，尤其是当类型推导依赖代码中很远处的信息时。例如

```cpp
auto foo = x.add_foo();
auto i = y.Find(key);
```

如果 `y` 的类型不是很明确，或者前面声明了很多行，则不容易看出返回类型。

* 不容易判断是引用还是复制。
* 如果将推导类型作为接口的一部分，那么在更改值时可能还更改了类型，从而导致激进的 API 更改。

使用方法：

* 使用类型推导只是为了使代码更清晰或更安全。



##### 函数模板参数推导

函数模板参数推导几乎都可行。类型推导是与函数模板交互的默认方式，因为它允许函数模板像多个重载函数一样工作。

```cpp
template <typename T>
void f(T t);

f(0);  // Invokes f<int>(0)
```



##### 局部变量类型推导

局部变量可以使用类型推导，使代码更清晰。例如

```cpp
std::unique_ptr<WidgetWithBellsAndWhistles> widget =
    std::make_unique<WidgetWithBellsAndWhistles>(arg1, arg2);
absl::flat_hash_map<std::string,
                    std::unique_ptr<WidgetWithBellsAndWhistles>>::const_iterator
    it = my_map_.find(key);
std::array<int, 6> numbers = {4, 8, 15, 16, 23, 42};
```

就可以简写为

```cpp
auto widget = std::make_unique<WidgetWithBellsAndWhistles>(arg1, arg2);
auto it = my_map_.find(key);
std::array numbers = {4, 8, 15, 16, 23, 42};
```

类型有时包含有用信息和目标的混合。在许多情况下，容器类型甚至 key 类型都不相关，但 value 的类型可能很有用。这种情况可以使用

```cpp
if (auto it = my_map_.find(key); it != my_map_.end()) {
  WidgetWithBellsAndWhistles& widget = *it->second;
  // Do stuff with `widget`
}
```

显式地定义局部变量，对迭代器使用类型推导。



##### 返回类型推导

仅当函数体的 `return` 语句数量非常少并且其它代码很少的时候，才使用返回类型推导。仅当函数或 lambda 的范围非常小的时候使用，因为具有推导返回类型的函数不顶用抽象边界：实现就是接口。特别是，头文件中的公共函数几乎不应该有推导的返回类型。



##### 参数类型推导

应当谨慎使用 lambda 的 `auto` 参数类型，因为实际类型由调用 lambda 的代码确定，而非 lambda 的定义确定。因此，显式类型总是更清晰，除非 lambda 被显式调用得非常接近它的定义位置，或者被传递给常见的接口。例如

```cpp
// Sort `vec` in decreasing order
std::sort(vec.begin(), vec.end(), [](auto lhs, auto rhs) { return lhs > rhs; });
```



##### Lambda 初始捕获

初始化捕获由更具体的样式规则（类模板参数推导）确定，该规则很大程度上取代了类型推导的一般规则。



##### 结构化绑定

结构化绑定可以通过为较大对象的元素提供有意义的名称来提供额外的信息。例如

```cpp
auto [iter, success] = my_map.insert({key, value});
if (!success) {
  iter->second = value;
}
```

不过注意除非 API 强制使用，否则不应该使用 pair 或 tuple 类型。



如果绑定的对象时结构，则提供于用法相关的名称可以更有帮助。如果它与绑定名称不匹配，建议使用注释

```cpp
auto [/*field_name1=*/bound_name1, /*field_name2=*/bound_name2] = ...
```

与函数参数注释一样，这可以使工具检测字段的顺序是否错误。



#### 类模板参数推导

仅对已明确选择支持类模板参数推导的模板使用。



类模板参数推导（CTAD）发生在使用命名模板的类型声明变量时，并且未提供模板参数列表。

```cpp
std::array a = {1, 2, 3};  // `a` is a std::array<int, 3>
```

这种推导可以是显式或隐式的。显式推导类似于具有尾随返回类型的函数声明

```cpp
namespace std {
template <class T, class... U>
array(T, U...) -> std::array<T, 1 + sizeof...(U)>;
}
```



优势：

* CTAD 有时可以让代码省略模板。

缺陷：

* 从构造函数生成的隐式推导可能不完全正确。
* 存在与 `auto` 相同的缺点。

使用方法：

* 不要将 CTAD 与给定模板一起使用，除非模板的维护者通过提供至少一个明确的推导来选择支持 CTAD 的使用。
* 必须遵循类型推导的一般规则。



#### 指定初始设定项

初始值设定项允许通过显式命名聚合的字段来初始化聚合

```cpp
struct Point {
    float x = 0.0;
    float y = 0.0;
    float z = 0.0;
};

Point p = {
    .x = 1.0,
    .y = 2.0,
    // z will be 0.0
};
```

显式列出的字段将按照指定方式进行初始化，而其它字段的初始化方式与传统聚合初始化表达式 `Point{1.0, 2.0}` 中的初始化方式相同。



优势：

* 指定的初始值设定项可以生成方便且高度可读的聚合表达式。

缺陷：

* 虽然指定的初始值设定项一直是 C 标准的一部分，但是 C++20 之前 C++ 不支持。
* C++ 标准中的规则比 C 和编译器扩展中的规则更严格，要求指定的初始值设定项的显示顺序与字段在结构定义中的显示顺序相同。

使用方法：

* 仅以符合 C++20 的形式使用指定的初始值设定项：初始值设定项的显示顺序与字段在结构定义中的显示顺序相同。



#### Lambda 表达式

在适当的情况下使用 lambda 表达式。当 lambda 将转义当前范围时，应当使用显式捕获。



优势：

* Lambda 比其它定义后传递给 STL 算法的函数对象方法更简洁易读。
* 适当使用默认捕获可以消除冗余。
* Lambda 与 `std::function,std::bind` 可以组合用作通用回调机制。

缺陷：

* Lambda 中的变量捕获可能是空指针的错误来源。
* 默认按值捕获可能产生误导，不能防止空指针错误。按值捕获指针不会导致深拷贝，因此与按引用捕获具有相同的生命周期问题。
* 捕获实际上声明了新的变量，但是难以被识别。
* 初始化捕获本质上依赖类型推导，存在许多与 auto 相同的缺点。

```cpp
std::unique_ptr<Foo> foo = ...;
[foo = std::move(foo)] () {
  ...
}

[foo = std::vector<int>({1, 2, 3})] () {
  ...
}
```

* Lambda 的使用可能会失控：嵌套过长的匿名函数会使代码难以理解。

使用方法：

* 适当情况下可以使用 lambda 表达式。如果 lambda 可能离开当前范围，首选显式捕获。例如

```cpp
{
  Foo foo;
  ...
  executor->Schedule([&] { Frobnicate(foo); })
  ...
}
// BAD! The fact that the lambda makes use of a reference to `foo` and
// possibly `this` (if `Frobnicate` is a member function) may not be
// apparent on a cursory inspection. If the lambda is invoked after
// the function returns, that would be bad, because both `foo`
// and the enclosing object could have been destroyed.
```

这种写法容易在 foo 销毁以后产生错误。正确的方式是

```cpp
{
  Foo foo;
  ...
  executor->Schedule([&foo] { Frobnicate(foo); })
  ...
}
// BETTER - The compile will fail if `Frobnicate` is a member
// function, and it's clearer that `foo` is dangerously captured by
// reference.
```

* 仅当 lambda 的生存期明显短于任何潜在的捕获时，才使用默认引用捕获 `&` 。
* 仅使用默认按值捕获 `=` 用于为短 lambda 绑定几个变量。出现在非静态类成员函数中并引用非静态类成员的 lambda 必须显式捕获 `this` 或用 `&` 捕获。
* 仅使用捕获来得到封闭范围中的变量。不要使用带有初始值设定项的捕获引入新的名称。



#### 模板元编程

避免复杂的模板编程。



优势：

* 模板元编程允许极其灵活的接口，它们具有类型安全和高性能。

缺陷：

* 模板元编程使用的技术晦涩难懂。以复杂方式使用模板的代码通常不可读，并且难以调试和维护。
* 模板元编程通常会导致极差的编译时错误信息。
* 大规模重构非常困难。



#### concept 和 requires

谨慎使用 `concept` 。避免在头文件引入新的 `concept`，除非它被标记为库的内部内容。不要定义编译器未强制执行的概念。比起模板元编程，更建议使用约束，并且避免使用 `template<Concept T>`，而使用 `requires(Concept<T>)` 。



优势：

* 概念允许编译器在涉及模板时生成更好的错误信息。
* 概念可以减少定义和使用编译时约束所需的模板，通常可以提高代码的清晰度。
* 约束提供了模板和 SFINAE 难以实现的技术。

缺陷：

* 概念会使代码更复杂难懂。
* 概念会增加代码耦合、刚性和僵化。
* 大规模重构非常困难。

使用方法：

* 存在等价概念时，应该优先使用标准库的预定义概念。例如 `std::integral` 应该在 C++20 中使用。
* 首选现代约束语法 `requires(Condition)`，避免使用遗留的模板元编程结构 `std::enable_if<Condition>` 以及 `template<Conecpt T>` 语法。
* 不要手动重新实现任何现有概念或特征。例如使用 `requires(std::default_initializable<T>)` 而不是 `requires(requires { T v; })` 。
* 新的 `concept` 声明应该很少，并且只能在库中定义，以便不会在 API 边界中公开。
* 不要定义与函数体重复的概念，也不要强加在阅读代码体或生成的错误消息时显而易见的 `requires`，例如

```cpp
template <typename T>     // Bad - redundant with negligible benefit
concept Addable = std::copyable<T> && requires(T a, T b) { a + b; };
template <Addable T>
T Add(T x, T y, T z) { return x + y + z; }
```

最好将代码保留为普通模板，除非使用概念可以带来重大改进。

* 不要使用任何主要好处来自语义约束的概念。



#### 别名

公共别名用于 API 用户的便利。



有几种方法创建别名

```cpp
using Bar = Foo;
typedef Foo Bar;  // But prefer `using` in C++ code.
using ::other_namespace::Foo;
using enum MyEnumType;  // Creates aliases for all enumerators in MyEnumType.
```

最好使用 `using` 而不是 `typedef`，它提供于其余部分更一致的语法，并且适用于模板。



优势：

* 别名可以通过简化长名称或复杂名称来提高可读性。
* 别名可以通过在一个位置命名 API 中重复使用的类型来减少重复。

缺陷：

* 别名会增加引用它的头文件中 API 的实体数量，从而增加复杂性。
* 可能造成命名冲突。

使用方法：

* 不要仅仅为了节省输入，在公共 API 中使用别名。
* 定义别名时，需要记录新名称的意义。
* 不要将命名空间别名放在公共 API 中。



#### Switch 语句

如果不是以枚举值为条件，`switch` 语句始终应具有 `default` 情形。如果没有默认情况，应该输出错误。例如

```cpp
switch (var) {
  case 0: {
    ...
    break;
  }
  case 1: {
    ...
    break;
  }
  default: {
    LOG(FATAL) << "Invalid value in switch statement: " << var;
  }
}
```



### 命名

#### 一般命名规则

使用描述对象的用途或意义的名称。描述性应当与名称的可见范围成正比。模板参数应遵循其类别的命名样式：

* 类型模板参数使用类型名称的规则；
* 非类型模板参数使用变量名称的规则；

下面展示了一组命名方式的对比：

```cpp
// 好的命名方式
class MyClass {
 public:
  int CountFooErrors(const std::vector<Foo>& foos) {
    int n = 0;  // Clear meaning given limited scope and context
    for (const auto& foo : foos) {
      ...
      ++n;
    }
    return n;
  }
  void DoSomethingImportant() {
    std::string fqdn = ...;  // Well-known abbreviation for Fully Qualified Domain Name
  }
 private:
  const int kMaxAllowedConnections = ...;  // Clear meaning within context
};

// 不好的命名方式
class MyClass {
 public:
  int CountFooErrors(const std::vector<Foo>& foos) {
    int total_number_of_foo_errors = 0;  // Overly verbose given limited scope and context
    for (int foo_index = 0; foo_index < foos.size(); ++foo_index) {  // Use idiomatic `i`
      ...
      ++total_number_of_foo_errors;
    }
    return total_number_of_foo_errors;
  }
  void DoSomethingImportant() {
    int cstmr_id = ...;  // Deletes internal letters
  }
 private:
  const int kNum = ...;  // Unclear meaning within broad scope
};
```



#### 文件名

文件名应全部小写，可以包含下划线或短划线。例如

- `my_useful_class.cc`
- `my-useful-class.cc`
- `myusefulclass.cc`
- `myusefulclass_test.cc `

C++ 文件应以 `.cc` 结尾，头文件以 `.h` 结尾。依赖于特定点，以文本方式包含的文件以 `.inc` 结尾。



#### 类型名

类型名以大写字母开头，每个新单词有一个大写字母，不带下划线。例如

```cpp
// classes and structs
class UrlTable { ...
class UrlTableTester { ...
struct UrlTableProperties { ...

// typedefs
typedef hash_map<UrlTableProperties *, std::string> PropertiesMap;

// using aliases
using PropertiesMap = hash_map<UrlTableProperties *, std::string>;

// enums
enum class UrlTableError { ...
```



#### 变量名

变量和数据成员的名称为小写，带有下划线。类（不是结构）的数据成员带有尾随下划线。例如

```cpp
a_local_variable, a_struct_data_member, a_class_data_member_
```



#### 公共变量名

```cpp
std::string table_name;  // OK - snake_case.
std::string tableName;   // Bad - mixed case.
```



#### 类数据成员

类数据成员（静态和非静态）的命名方式与普通的非成员变量类似，但是带有尾部下划线。

```cpp
class TableInfo {
  ...
 private:
  std::string table_name_;  		// OK - underscore at end.
  static Pool<TableInfo>* pool_;  	// OK.
};
```



#### 结构数据成员

结构的数据成员，包括静态和非静态，命名方式与普通的非成员变量类似。

```cpp
struct UrlTableProperties {
  std::string name;
  int num_entries;
  static Pool<UrlTableProperties>* pool;
};
```



#### 常量名

声明 `constexpr` 或 `const` 变量，其值在程序持续时间内不变，以 `k` 作为前缀，后跟大小写混合。极少数情况下，可以用下划线分隔，不使用大写分隔。例如

```cpp
const int kDaysInAWeek = 7;
const int kAndroid8_0_0 = 24;  // Android 8.0.0
```

所有具有静态持续时间的变量都以这种方式命名，包括模板的不同实例可能具有不同值的变量。对于其它储存类的变量，此约定可选。例如

```cpp
void ComputeFoo(absl::string_view suffix) {
  // 可选是否遵循常量名约定
  const absl::string_view kPrefix = "prefix";
  const absl::string_view prefix = "prefix";
  ...
}

void ComputeFoo(absl::string_view suffix) {
  // 变量值随着函数调用的不同而改变，不应当使用常量名约定
  const std::string kCombined = absl::StrCat(kPrefix, suffix);
  ...
}
```



#### 函数名

通常函数以大写字母开头，每个单词首字母大写。

```cpp
AddTableEntry()
DeleteUrl()
OpenFileOrDie()
```

访问器和赋值器可以像变量一样命名，例如

```cpp
int count();
void set_count(int count);
```



#### 命名空间名

命名空间名均小写，单词之间用下划线分隔。顶层命名空间名称基于项目名称。避免嵌套命名空间与已知顶层命名空间冲突。



#### 枚举名

枚举应当像常量一样命名，而不是宏。例如

```cpp
enum class UrlTableError {
  kOk = 0,
  kOutOfMemory,
  kMalformedInput,
};
```



#### 宏名

通常不应使用宏。如果绝对需要它们，应使用全大写字母和下划线以及项目前缀命名。例如

```cpp
#define MYPROJECT_ROUND(x) ...
```



#### 例外情况

如果要命名类似于现有 C 或 C++ 实体的内容，可以遵循现有的命名约定方案。

```cpp
// 根据 open() 的形式命名
bigopen()
// typedef
uint
// 根据 pos 的形式命名
bigpos
// 根据 STL 的形式命名
sparse_hash_map
// 根据 INT_MAX 命名的常量
LONGLONG_MAX
```
